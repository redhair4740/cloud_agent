# 边缘数据同步实现评审

来源：落实本轮 reviewer 输出，核对 `sync` 子域当前后端/前端实现是否满足“实时同步、冲突处理、对象级快照、运维可见性”要求。

- 当前状态：5 项整改代码已落地，真实编译/前后端联调/EMQX 实链路验证未完成，因此文档状态暂保留“未完成”。

- [x] `S1(P1)` 发布同步命令时必须携带对象实际内容
  - 位置：`moon_cloud_backend/vmesh-module-edge/src/main/java/com/wf/vmesh/module/edge/service/sync/EdgeSyncServiceImpl.java:507-515`
  - 真实证据：
    - 当前主工作区的 [buildSyncPayload](/mnt/e/projects/ai/moon/moon_cloud_backend/vmesh-module-edge/src/main/java/com/wf/vmesh/module/edge/service/sync/EdgeSyncServiceImpl.java:635) 已补齐 `object_key`、`version`、`hash`、`content`。
    - [publish](/mnt/e/projects/ai/moon/moon_cloud_backend/vmesh-module-edge/src/main/java/com/wf/vmesh/module/edge/service/sync/EdgeSyncServiceImpl.java:177) 已按对象级拆分明细；[retryItem](/mnt/e/projects/ai/moon/moon_cloud_backend/vmesh-module-edge/src/main/java/com/wf/vmesh/module/edge/service/sync/EdgeSyncServiceImpl.java:245) 与 [scanRetryWaitingItems](/mnt/e/projects/ai/moon/moon_cloud_backend/vmesh-module-edge/src/main/java/com/wf/vmesh/module/edge/service/sync/EdgeSyncServiceImpl.java:397) 都复用 `commandPayloadJson` / descriptor 重建真实载荷。
  - 问题：review 指出的“空壳同步消息”已完成代码整改，但尚未验证边端是否按新 payload 消费。
  - 完成标准：同步消息必须按对象类型携带实际内容，至少包含对象标识、版本/哈希、业务字段本体；`publish()` 与 `retryItem()` 都要复用真实对象载荷而不是空壳元数据。
  - 完成说明：已完成代码整改；未验证边端消费链路与真实 MQTT 收敛。

- [x] `S1(P1)` 冲突确认后不要只改状态而不推进真实收敛
  - 位置：`moon_cloud_backend/vmesh-module-edge/src/main/java/com/wf/vmesh/module/edge/service/sync/EdgeSyncServiceImpl.java:253-307`
  - 真实证据：
    - [confirmCloud](/mnt/e/projects/ai/moon/moon_cloud_backend/vmesh-module-edge/src/main/java/com/wf/vmesh/module/edge/service/sync/EdgeSyncServiceImpl.java:282) 现在会重建云端对象 descriptor，并通过 [republishConflictObject](/mnt/e/projects/ai/moon/moon_cloud_backend/vmesh-module-edge/src/main/java/com/wf/vmesh/module/edge/service/sync/EdgeSyncServiceImpl.java:548) 重新下发真实对象。
    - 当前实现已切换为“节点级云边同步”：`sync.conflict` 对节点级对象不再允许“采纳边端回写”，而是保留 diff 快照后按云端基线自动重发；`confirm-cloud` 仅保留为历史冲突或人工重发入口。
    - 冲突重发后的状态推进继续收口到 [markSnapshotInSync](/mnt/e/projects/ai/moon/moon_cloud_backend/vmesh-module-edge/src/main/java/com/wf/vmesh/module/edge/service/sync/EdgeSyncServiceImpl.java:725)、[refreshBatchStatus](/mnt/e/projects/ai/moon/moon_cloud_backend/vmesh-module-edge/src/main/java/com/wf/vmesh/module/edge/service/sync/EdgeSyncServiceImpl.java:527)。
  - 问题：review 指出的“只改状态不收敛”已完成代码整改，但尚未验证真实冲突确认链路。
  - 完成标准：节点级对象冲突必须触发真实云到边重同步，并推进 `edge_sync_item`、重算批次状态，保证冲突处理后运维列表闭环；不再保留“采纳边端回写”路径。
  - 完成说明：已完成代码整改；未验证真实冲突自动重发和 EMQX 重发链路。

- [x] `S1(P1)` 同步快照必须按 `objectKey` 维度保存和查询
  - 位置：`moon_cloud_backend/vmesh-module-edge/src/main/java/com/wf/vmesh/module/edge/dal/mapper/sync/EdgeSyncSnapshotMapper.java:21-26`
  - 真实证据：
    - review 原问题：旧实现的 snapshot 查询只按 `nodeId + objectType` 命中，`upsertSnapshot()` / `createConflict()` / `findItemByBatchAndObjectType()` 都会忽略 `objectKey`。
    - 当前整改后已按 `objectKey` 收口到 snapshot 查询、diff 查询与 item 匹配链路。
  - 问题：同一节点下多个同类型对象会互相覆盖，`/diff/{objectType}` 也无法定位具体是哪一个对象产生差异。
  - 完成标准：快照、冲突、明细匹配都必须引入 `objectKey` 维度；查询接口与前端详情也要能按对象键查看差异。
  - 完成说明：已完成代码调整；主工作区已把 `getDiff`、snapshot 分页、snapshot upsert、conflict 创建、item 匹配统一改为按 `objectKey` 收口。未验证多对象运行态。

- [x] `S2(P2)` 前端应将 `RESOLVED_CLOUD/RESOLVED_EDGE` 视为已解决状态
  - 位置：`moon_cloud_frontend/src/views/edge/sync/index.vue:128-150`
  - 真实证据：
    - review 原问题：旧页面只把 `RESOLVED` 视为已解决，导致 `RESOLVED_CLOUD/RESOLVED_EDGE` 仍显示“待确认”并继续暴露确认按钮。
    - 当前整改后，前端已把 `RESOLVED_CLOUD`、`RESOLVED_EDGE` 纳入已解决集合，并在确认后刷新冲突列表。
  - 问题：用户处理完冲突后，页面仍显示“待确认”并继续暴露两个确认按钮；再次点击会命中后端“已解决”异常。
  - 完成标准：前端状态映射必须把 `RESOLVED_CLOUD`、`RESOLVED_EDGE` 纳入已解决集合，并同步修正标签、按钮显隐和筛选口径。
  - 完成说明：已完成代码调整；主工作区已补齐冲突状态映射、标签和按钮显隐，并在确认后刷新 overview/batch/conflict。未执行前端构建或页面验证。

- [x] `S2(P2)` `checkBatchCompletion()` 不应把 `CONFLICT_PENDING` 直接视为批次已完成
  - 位置：`moon_cloud_backend/vmesh-module-edge/src/main/java/com/wf/vmesh/module/edge/service/sync/EdgeSyncServiceImpl.java:433-450`
  - 真实证据：
    - review 原问题：旧实现会把 `CONFLICT_PENDING` 也纳入“可完成”判断，导致冲突未处理的批次被错误写成 `COMPLETED`。
    - 当前整改后，批次状态已重构为 [refreshBatchStatus](/mnt/e/projects/ai/moon/moon_cloud_backend/vmesh-module-edge/src/main/java/com/wf/vmesh/module/edge/service/sync/EdgeSyncServiceImpl.java:507)，冲突未确认时固定落到 `WAITING_CONFIRMATION`。
  - 问题：会出现“批次已完成”但系统仍存在 `CONFLICT_PENDING` 的不一致状态，和“冲突待确认”语义冲突，容易误导运维与业务判断。
  - 完成标准：批次完成判定需要把“自动链路完成”和“人工确认完成”拆开；在冲突未处理前，批次不应直接标记为 `COMPLETED`，除非明确新增独立状态字段承载这层语义。
  - 完成说明：已完成代码调整；主工作区已把冲突未确认批次改为 `WAITING_CONFIRMATION`，前端也已补齐对应展示口径。未验证运行态状态机。
