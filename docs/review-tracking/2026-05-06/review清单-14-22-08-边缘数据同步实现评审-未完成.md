# 边缘数据同步实现评审

来源：落实本轮 reviewer 输出，核对 `sync` 子域当前后端/前端实现是否满足“实时同步、冲突处理、对象级快照、运维可见性”要求。

- [ ] `S1(P1)` 发布同步命令时必须携带对象实际内容
  - 位置：`moon_cloud_backend/vmesh-module-edge/src/main/java/com/wf/vmesh/module/edge/service/sync/EdgeSyncServiceImpl.java:507-515`
  - 真实证据：
    - 当前 [buildSyncPayload](/mnt/e/projects/AI/moon/moon_cloud_backend/vmesh-module-edge/src/main/java/com/wf/vmesh/module/edge/service/sync/EdgeSyncServiceImpl.java:507) 只写入 `message_id`、`occurred_at`、`batch_key`、`object_type`、`node_id`、`device_id`。
    - `scanRetryWaitingItems()` 在重发时也会回退复用这份 payload [EdgeSyncServiceImpl.java:404-406](/mnt/e/projects/AI/moon/moon_cloud_backend/vmesh-module-edge/src/main/java/com/wf/vmesh/module/edge/service/sync/EdgeSyncServiceImpl.java:404)。
  - 问题：下行同步消息只有元数据，没有对象版本、哈希或业务字段本体；边端即使收到消息，也拿不到需要应用的配置/资源/元数据内容。
  - 完成标准：同步消息必须按对象类型携带实际内容，至少包含对象标识、版本/哈希、业务字段本体；`publish()` 与 `retryItem()` 都要复用真实对象载荷而不是空壳元数据。
  - 完成说明：未完成；本轮只完成静态审查，未验证边端消费链路。

- [ ] `S1(P1)` 冲突确认后不要只改状态而不推进真实收敛
  - 位置：`moon_cloud_backend/vmesh-module-edge/src/main/java/com/wf/vmesh/module/edge/service/sync/EdgeSyncServiceImpl.java:253-307`
  - 真实证据：
    - [confirmCloud](/mnt/e/projects/AI/moon/moon_cloud_backend/vmesh-module-edge/src/main/java/com/wf/vmesh/module/edge/service/sync/EdgeSyncServiceImpl.java:253) 和 [confirmEdge](/mnt/e/projects/AI/moon/moon_cloud_backend/vmesh-module-edge/src/main/java/com/wf/vmesh/module/edge/service/sync/EdgeSyncServiceImpl.java:282) 只更新 `edge_sync_conflict.status/resolved*` 与 `edge_sync_snapshot.syncStatus`。
    - 这两个方法没有重新下发“以云端为准”的对象，也没有把“采纳边端回写”落回云端数据。
    - 对应 `edge_sync_item` 也没有从 `CONFLICT_PENDING` 推进，`checkBatchCompletion()` 没有被重新触发。
  - 问题：用户点击“确认”后数据实际上没有收敛，且批次会长期停留在未完成状态。
  - 完成标准：`confirmCloud` 必须触发真实云到边重同步；`confirmEdge` 必须触发真实边到云收敛；两者都要推进 `edge_sync_item`、重算批次状态，并保证冲突处理后运维列表闭环。
  - 完成说明：未完成；本轮只完成静态审查，未验证真实冲突收敛链路。

- [ ] `S1(P1)` 同步快照必须按 `objectKey` 维度保存和查询
  - 位置：`moon_cloud_backend/vmesh-module-edge/src/main/java/com/wf/vmesh/module/edge/dal/mapper/sync/EdgeSyncSnapshotMapper.java:21-26`
  - 真实证据：
    - 当前 [selectByNodeAndObjectType](/mnt/e/projects/AI/moon/moon_cloud_backend/vmesh-module-edge/src/main/java/com/wf/vmesh/module/edge/dal/mapper/sync/EdgeSyncSnapshotMapper.java:21) 只按 `nodeId + objectType` 查询。
    - `upsertSnapshot()` 与 `createConflict()` 也都依赖这条查询 [EdgeSyncServiceImpl.java:455-489](/mnt/e/projects/AI/moon/moon_cloud_backend/vmesh-module-edge/src/main/java/com/wf/vmesh/module/edge/service/sync/EdgeSyncServiceImpl.java:455)。
    - `findItemByBatchAndObjectType()` 同样忽略 `objectKey`，只按 `objectType` 命中第一条明细 [EdgeSyncServiceImpl.java:425-430](/mnt/e/projects/AI/moon/moon_cloud_backend/vmesh-module-edge/src/main/java/com/wf/vmesh/module/edge/service/sync/EdgeSyncServiceImpl.java:425)。
  - 问题：同一节点下多个同类型对象会互相覆盖，`/diff/{objectType}` 也无法定位具体是哪一个对象产生差异。
  - 完成标准：快照、冲突、明细匹配都必须引入 `objectKey` 维度；查询接口与前端详情也要能按对象键查看差异。
  - 完成说明：未完成；本轮只完成静态审查，未验证多对象场景。

- [ ] `S2(P2)` 前端应将 `RESOLVED_CLOUD/RESOLVED_EDGE` 视为已解决状态
  - 位置：`moon_cloud_frontend/src/views/edge/sync/index.vue:128-150`
  - 真实证据：
    - 当前页面只把 `row.status === 'RESOLVED'` 渲染为“已解决” [index.vue:130-131](/mnt/e/projects/AI/moon/moon_cloud_frontend/src/views/edge/sync/index.vue:130)。
    - 两个确认按钮也只在 `row.status !== 'RESOLVED'` 时隐藏 [index.vue:141-149](/mnt/e/projects/AI/moon/moon_cloud_frontend/src/views/edge/sync/index.vue:141)。
    - 后端确认接口返回的是 `RESOLVED_CLOUD` 和 `RESOLVED_EDGE` [EdgeSyncServiceImpl.java:263-264](/mnt/e/projects/AI/moon/moon_cloud_backend/vmesh-module-edge/src/main/java/com/wf/vmesh/module/edge/service/sync/EdgeSyncServiceImpl.java:263)、[EdgeSyncServiceImpl.java:292-293](/mnt/e/projects/AI/moon/moon_cloud_backend/vmesh-module-edge/src/main/java/com/wf/vmesh/module/edge/service/sync/EdgeSyncServiceImpl.java:292)。
  - 问题：用户处理完冲突后，页面仍显示“待确认”并继续暴露两个确认按钮；再次点击会命中后端“已解决”异常。
  - 完成标准：前端状态映射必须把 `RESOLVED_CLOUD`、`RESOLVED_EDGE` 纳入已解决集合，并同步修正标签、按钮显隐和筛选口径。
  - 完成说明：未完成；本轮只完成静态审查，未执行前端构建或页面验证。
