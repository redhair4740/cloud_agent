## 仍需实施清单

- [x] pnpm 单一路线收口
  - 目标：让本地、CI、手工发布全部使用同一套 pnpm 安装与构建链路，避免 `npm` / `pnpm` 混用导致依赖树漂移
  - 计划动作：
    - 在 `package.json` 增加明确的 `packageManager` 字段，例如 `pnpm@8.x` 或团队确认版本
    - 生成并保留 `pnpm-lock.yaml` 作为唯一锁文件
    - 移除发布脚本里对 `package-lock.json` 的文件存在校验
    - 将 `deploy/Jenkinsfile` 的 `npm ci` / `npm run build:prod` 调整为 pnpm 对应命令
    - 将 `deploy/manual-release.sh`、`deploy/manual-release.ps1` 的工具检查、dry-run 输出、真实构建命令全部调整为 pnpm 对应命令
    - 复核 `package.json` 中 `clean`、`clean:cache` 是否继续使用 `npx`，优先改为 `pnpm exec rimraf ...`
  - 风险关注：
    - 当前仓内未发现 `pnpm-lock.yaml`，直接改为 `pnpm install --frozen-lockfile` 会因为锁文件缺失失败
    - CI 节点需要确认 pnpm 安装方式，例如 Corepack 或预装 pnpm
  - 完成标准：本地说明、Jenkins、手工发布脚本全部只表达 pnpm 路线；锁文件存在且作为唯一依赖锁
  - 完成说明：已新增 `packageManager: pnpm@8.15.9`，`clean` / `clean:cache` 已改为 `pnpm exec rimraf`，Jenkins 与手工发布脚本已改为 pnpm；`manual-release.sh#load_config` 已改为逐行读取并去除 CRLF 行尾，避免端口值带 `\r` 导致数字校验失败；剩余 `pnpm-lock.yaml` 需通过真实 `pnpm install` 生成，当前未执行。
  - 验证状态：未验证

- [x] 精简 Vite 默认插件链
  - 目标：降低 `pnpm dev` 冷启动、依赖扫描和生产构建链路开销
  - 计划动作：
    - 开发态默认不启用 `vite-plugin-eslint`，把 ESLint 收口到显式 lint 命令或 CI 阶段
    - 复核项目是否仍需要 `vite-plugin-top-level-await`，无真实依赖后移除插件和依赖
    - 仅在生产构建启用 `vite-plugin-compression`
    - 仅在开发态或明确需要时启用 `vite-plugin-progress`
    - 收缩 `AutoImport` 与 `Components` 扫描范围，避免非必要目录被持续扫描
  - 风险关注：
    - 移除 `topLevelAwait` 前必须确认构建产物或依赖中没有依赖该转换能力
    - 调整 `AutoImport` / `Components` 后必须回归全局组件、Element Plus 自动导入和类型声明生成
  - 完成标准：插件按 `command` / `mode` 条件启用，非必要插件不进入默认 dev 链路
  - 完成说明：`createVitePlugins` 已按 `command` / `mode` 条件启用插件；dev 默认不启用 ESLint，压缩仅 prod build 启用，progress 仅非 build 启用，`vite-plugin-top-level-await` 已移除。
  - 验证状态：未验证

- [x] 收缩 `optimizeDeps`
  - 目标：减少首次 `pnpm dev` 预构建成本，避免长白名单拖慢冷启动
  - 计划动作：
    - 移除大部分手工 `include` 白名单，只保留经真实报错证明必须预构建的依赖
    - 去掉 `element-plus/es/components/*/style/css` 这类手工样式预构建项
    - 保留最小 `exclude` 策略，例如 `@iconify/json`
    - 对 `markmap-*`、`@wangeditor-next/*`、`echarts-*` 等非首屏依赖，优先避免放入默认预构建白名单
  - 风险关注：
    - 收缩后可能暴露历史被白名单掩盖的 optimized dependency 问题
    - 需要用相同机器、相同缓存清理口径采集冷启动对比
  - 完成标准：`build/vite/optimize.ts` 从长白名单收敛为最小集合，并有真实启动输出支撑
  - 完成说明：`build/vite/optimize.ts` 已收缩为基础常用依赖，已移除 Element Plus 样式项、markmap、wangeditor、echarts 等非首屏重依赖预构建。
  - 验证状态：未验证

- [x] 主入口按需装载重功能
  - 目标：避免所有用户在首屏阶段承担表单设计、打印等低频能力的初始化成本
  - 计划动作：
    - 从 `src/main.ts` 移除默认全局 `setupFormCreate(app)` 注册
    - 为 `form-create` / `@form-create/designer` 提供幂等按需安装入口
    - 从 `src/main.ts` 移除默认全局 `app.use(print)` 注册，改为打印相关页面或指令按需装载
    - `wangEditor` 维持组件级跟踪，不再作为主入口直接注册项处理
  - 风险关注：
    - 按需注册必须幂等，避免多次进入页面重复安装插件
    - 表单构建器、BPM 表单、打印模板等低频但关键页面必须做功能回归
  - 完成标准：普通首屏不再执行表单设计器和打印插件的全局初始化；相关页面首次进入时能力可正常使用
  - 完成说明：`src/main.ts` 已移除 `form-create` 与打印插件默认全局注册；`setupFormCreate` 已幂等；`ensurePrint` 使用动态导入；`infra/build` 页面已按需安装表单设计器。
  - 验证状态：未验证

- [x] 依赖治理
  - 目标：删除高概率冗余依赖，修正运行期依赖分类，避免误删真实业务依赖
  - 计划动作：
    - 保留 `bpmn-js`，当前存在 BPMN 设计器 / 查看器真实 active import
    - 优先复核并候选删除 `xml-js`，当前仅见注释引用，实际 XML 解析已见 `steady-xml`
    - 优先复核并候选删除 `bpmn-js-properties-panel`，当前仅见注释引用
    - 复核 `markmap-common` 是否仅为传递依赖或历史残留；若源码无 direct import 且锁文件验证通过，再删除直接依赖声明
    - 复核 `camunda-bpmn-moddle` 是否仍支撑 BPMN Camunda 扩展；若无 direct import 且功能不需要，再删除
    - 将真实运行期直接引用的 BPMN 相关依赖放入 `dependencies`，不要放在 `devDependencies`
  - 风险关注：
    - 删除依赖前必须生成 pnpm 锁文件并执行安装 / 构建 / 关键页面回归
    - BPMN、思维导图、富文本等低频页面容易被常规首屏测试遗漏
  - 完成标准：候选依赖有“删除 / 保留”证据，删除项通过构建与关键页面回归
  - 完成说明：已删除 `xml-js`、`bpmn-js-properties-panel`、`markmap-common`、`camunda-bpmn-moddle`、`vite-plugin-top-level-await` 依赖声明；`bpmn-js` 已从 `devDependencies` 调整到 `dependencies`；BPMN 文件中的失效注释引用已清理。
  - 验证状态：未验证
