# Review 与优化建议跟踪

## 使用约定

- 适用范围：代码 review 结论、联调发现、优化建议、回归风险修复项。
- 记录方式：每个待处理事项都使用 Markdown 勾选框，未完成用 `[ ]`，完成后改成 `[x]`。
- 更新规则：每次 review 按日期创建独立文件夹（`yyyy-MM-dd`），按时间创建独立清单文件，命名格式为 `review清单-hh-mm-ss-文档名称-状态.md`，对应代码完成后，同步更新对应清单中的勾选状态、完成说明和文件名状态后缀。
- 命名约定：`文档名称` 使用简洁可读的中文/英文标题，避免 `/` 等文件系统保留字符；`状态` 仅允许 `已完成`、`未完成` 两种。
- 状态判定：清单中仍存在需要处理且未完成的 `[ ]` 项时，文件名标记为 `未完成`；若清单中不存在待处理项，或事项无需继续处理且已闭环，则标记为 `已完成`。
- 证据要求：每条任务尽量保留真实路径、方法名、字段名或脚本位置，方便复查。
- 验证口径：未执行真实验证时明确写"未验证"，不要把静态判断写成"已完成"。
