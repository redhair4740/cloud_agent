# 边缘设备上报方案

> 边缘端通过 `device.report` 事件上报设备清单，云端消费后维护设备台账运行态，管理面只读观测 + 元数据编辑。

## 1. 背景

### 需求来源

边缘节点下挂载的物理设备（相机、光源、传感器等）由边缘端 Agent 直接管理和发现。云端不管控设备生命周期，设备清单完全由边缘端定期上报。

### 当前现状

| 层 | 现状 | 证据 |
|---|------|------|
| **设备表** | [EdgeDeviceDO](../../../moon_cloud_backend/vmesh-module-edge/src/main/java/com/wf/vmesh/module/edge/dal/dataobject/device/EdgeDeviceDO.java) 已存在，含 deviceCode/name/type/edgeNodeId/status 等字段 | L1-L97 |
| **CRUD 接口** | [EdgeDeviceController](../../../moon_cloud_backend/vmesh-module-edge/src/main/java/com/wf/vmesh/module/edge/controller/admin/device/EdgeDeviceController.java) 已实现 create/update/delete/get/page/export | L1-L80+ |
| **Topic 路由** | [resolveEventTopicSuffix](../../../moon_cloud_backend/vmesh-module-edge/src/main/java/com/wf/vmesh/module/edge/service/runtime/EdgeTopicService.java#L86-L98) 已映射 `"device.report" -> "/telemetry/device-report"` | L90 |
| **ingest 消费** | [updateSnapshot 的 switch](../../../moon_cloud_backend/vmesh-module-edge/src/main/java/com/wf/vmesh/module/edge/service/runtime/EdgeRuntimeServiceImpl.java#L371-L410) 中 `device.report` 走 default 空分支，**未实现** | L409 |
| **测试 Payload** | [03-设备快照上报.json](../../test-data/edge-node-mqtt/03-设备快照上报.json) 已定义，devices 数组含 device_code/name/type/status/stream_status 等字段 | L1-L23 |
| **状态枚举** | [EdgeDeviceStatusEnum](../../../moon_cloud_backend/vmesh-module-edge/src/main/java/com/wf/vmesh/module/edge/enums/device/EdgeDeviceStatusEnum.java) 已定义 ONLINE/OFFLINE/RUNNING/WARNING 四态 | L1-L74 |

**结论：设备表的管理面（CRUD）已就绪，但运行态接入面（device.report 消费）为空。**

### 触发原因

`device.report` 是 EdgeRuntimeServiceImpl ingest 链路中最后一个未实现的 eventType 分支，需要补齐以完成边云设备数据闭环。

## 2. 目标

| 编号 | 目标 | 验收指标 |
|------|------|----------|
| G-1 | 边缘端通过 MQTT 上报设备清单，云端正确消费并写入 edge_device 表 | device.report 事件可触发 upsert，设备出现在列表页 |
| G-2 | 设备数据双层分离：运行态字段由边缘端写入，管理态字段由云端编辑 | 运行态字段在上报时覆盖，管理态字段不受上报影响 |
| G-3 | 设备移除后逻辑归档，支持重新启用 | 归档设备不出现在默认列表，恢复后重新出现 |
| G-4 | 以 (edgeNodeId, deviceCode) 为业务唯一键，拒绝跨节点漂移 | 同一 deviceCode 出现不同节点时记录告警并跳过 |
| G-5 | 上报频率可控，对 EMQX 和数据库压力在可接受范围内 | 500 节点 × 15 设备场景下日写入量 < 300 万次 |

## 3. 方案

### 3.1 总体方案：边缘端为真源、云端为消费者 + 元数据管理者

核心设计原则：

> **边缘端是设备数据的唯一写入源（Writer），云端是只读消费者（Reader）+ 管理面元数据编辑者（Metadata Editor）。**

#### 数据双层模型

```
┌─────────────────────────────────────────────┐
│              设备数据双层模型                  │
├──────────────────┬──────────────────────────┤
│  运行态层（Edge写） │  管理态层（Cloud 写）      │
│  ─────────────── │  ──────────────────────  │
│  status          │  name（显示名）            │
│  runtime_status  │  group_id（分组）          │
│  stream_status   │  region（区域）            │
│  runtime_attrs*  │  location（安装位置）       │
│  reported_at     │  install_time             │
│  offline_since   │  manufacturer / model     │
│  last_heartbeat_time │ resolution / serial  │
└──────────────────┴──────────────────────────┘
```

#### 关键决策结论

| # | 决策项 | 结论 | 落地含义 |
|---|--------|------|----------|
| D1 | 云端手动录入设备 | **不需要** | 禁用 create 接口或标记 deprecated；所有设备来自边缘端首次上报 |
| D2 | 设备被移除后的处理 | **逻辑删除 + 支持重新启用** | deleted 标记 + archivedAt 时间戳；提供 restore 接口清除标记 |
| D3 | 跨节点 deviceCode 处理 | **拒绝漂移** | (edgeNodeId, deviceCode) 联合唯一约束；冲突时记录告警日志并跳过 |
| D4 | 上报频率策略 | **增量驱动 + 定期全量兜底** | 变更即时上报 + 每 5 分钟全量快照对账 |

### 3.2 上报频率与压力控制

#### 压力来源拆解

```
边缘端 ──MQTT发布──▶ EMQX ──HTTP Sink──▶ 后端ingest ──▶ DB写入
         ↑               ↑                ↑              ↑
      发布QoS/频率    消息堆积/转发     并发HTTP请求     UPSERT行锁
```

| 压力点 | 全量定时上报（30s）的风险 | 优化策略 |
|--------|------------------------|----------|
| **EMQX 吞吐** | 实际风险低，EMQX 单节点支撑 10万+ msg/s | 无需特殊处理 |
| **后端 ingest 并发** | 中等风险，高峰期大量并发 HTTP POST 占用线程池 | 短窗口去重合并 |
| **数据库写入** | 主要风险，N 台设备 = N 次 INSERT/UPDATE | 批量 upsert + 频率控制 |
| **存储膨胀** | EdgeRuntimeEventDO 每次存完整 payload JSON | 事件表裁剪存储 |

#### 推荐上报策略：事件驱动 + 定期全量兜底

区分"常态"和"变态"，只在变化时即时通知，常态用低频全量兜底：

```
┌──────────────────────────────────────────────────────┐
│                  边缘端上报策略                         │
├──────────────────────┬───────────────────────────────┤
│  场景                 │  行为                          │
│  ─────────────────    │  ─────────────────────────    │
│  设备上线/发现         │  立即上报该设备（增量）          │
│  设备离线/断开         │  立即上报该设备状态变更（增量）   │
│  设备运行属性变化       │  立即上报该设备（增量）          │
│                      │                               │
│  定时全量快照（兜底）    │  每 5 分钟上报一次完整设备列表    │
│  （防止丢消息导致状态不准）│                               │
└──────────────────────┴───────────────────────────────┘
```

#### 参数建议

| 参数 | 推荐值 | 说明 |
|------|--------|------|
| 增量上报触发条件 | 设备增删、status 变化、stream_status 变化、关键指标超阈值 | 边缘端本地 diff 检测 |
| 全量快照间隔 | **300s（5 分钟）** | 兜底对账，不需要太频繁 |
| 单次上报最大设备数 | 无硬限，但建议单节点 < 200 台 | 超出考虑分批 |
| payload 大小控制 | devices 数组精简字段，避免携带大段日志/图像数据 | 只传状态相关字段 |

#### 三种模式量化估算对比

假设规模：**500 个边缘节点，平均每节点 15 台设备**

| 模式 | 上报频率 | 日消息量 | 日 DB 写入量 | 适用性 |
|------|----------|----------|-------------|--------|
| A. 全量 30s | 每节点 30s 一次 | ~144 万条/日 | ~2160 万次 upsert | ❌ 浪费大，压力大 |
| B. 全量 5min | 每节点 5min 一次 | ~14.4 万条/日 | ~216 万次 upsert | ✅ 可接受 |
| C. 增量 + 5min 全量 | 变更即时 + 5min 全量 | ~5-20 万条/日（含增量） | ~75-300 万次 upsert | ✅✅ **推荐** |

模式 C 在设备稳定运行时流量接近模式 B；设备频繁上下线时增量消息增多但均为有价值的变更通知。

#### 云端消费侧防抖设计

即使边缘端已做频率控制，云端仍需自我保护：

1. **dedupeKey 幂等** — 已有机制，同一 (nodeId, messageId) 不重复处理
2. **短窗口去重合并** — 用内存缓存记录 `(nodeId, deviceCode)` 最后同步版本号/时间戳；若本次 reportedAt ≤ 缓存中的 lastReportedAt 且差值在 10s 内则跳过
3. **批量 upsert 替代逐条** — 解析 devices[] 后批量 INSERT ... ON CONFLICT DO UPDATE，减少数据库往返
4. **原始事件裁剪存储** — EdgeRuntimeEventDO 的 payloadJson 只存摘要信息，完整 devices[] 已落入 edge_device 表

### 3.3 设备生命周期状态机

```
                    首次上报 ──────────▶ ┌──────────────┐
                                        │   ACTIVE     │◀── 正常运行
                                        └──────┬───────┘
                                               │
                                  ┌────────────┼────────────┐
                                  ▼            ▼            ▼
                           ┌──────────┐  ┌──────────┐  ┌──────────┐
                           │ OFFLINE  │  │ ABNORMAL │  │ ARCHIVED  │
                           └──────────┘  └──────────┘  └────┬─────┘
                           连续 N 期      上报异常属性        ▲
                           未上报                        │
                                                        │ 重新启用
                                                        ┘
```

状态说明：

| 状态 | 触发条件 | 行为 |
|------|----------|------|
| **ACTIVE** | 边缘端首次上报该 deviceCode | 新建设备记录，source=EDGE_REPORT |
| **OFFLINE** | 连续 N 次全量快照中未出现该设备 | 更新 offline_since，不影响归档历史 |
| **ABNORMAL** | 上报的 status/stream_status 异常（如 stream_lost、frame_drop） | 记录异常详情到 runtime_attrs，联动监控告警 |
| **ARCHIVED** | 云端手动归档 / 自动归档策略触发 | 逻辑删除（deleted=1），保留历史关联不悬空 |

> 不再使用 PRE_CREATED 状态，所有设备生成都来自边缘端首次上报。

### 3.4 数据模型变更

#### EdgeDeviceDO 扩展字段

基于现有 [EdgeDeviceDO](../../../moon_cloud_backend/vmesh-module-edge/src/main/java/com/wf/vmesh/module/edge/dal/dataobject/device/EdgeDeviceDO.java)，新增以下字段：

| 字段 | 类型 | 默认值 | 所属层 | 说明 |
|------|------|--------|--------|------|
| `source` | VARCHAR(32) | 'EDGE_REPORT' | 管理态 | 数据来源固定值，未来可扩展 |
| `reportedAt` | LocalDateTime | NULL | 运行态 | 最后上报时间 |
| `runtimeStatus` | VARCHAR(32) | NULL | 运行态 | 运行态在线状态（online/offline），区别于管理态 status 枚举 |
| `streamStatus` | VARCHAR(32) | NULL | 运行态 | 流媒体状态（running/stopped/stream_lost/frame_drop） |
| `runtimeAttrs` | JSONB | '{}' | 运行态 | 设备运行时动态属性（brightness, last_frame_at 等，因设备类型而异） |
| `offlineSince` | LocalDateTime | NULL | 运行态 | 开始离线的时间，用于判断是否应自动归档 |
| `archivedAt` | LocalDateTime | NULL | 管理态 | 归档时间，NULL 表示未归档 |

#### 联合唯一约束

```sql
ALTER TABLE edge_device
    ADD CONSTRAINT uk_edge_device_node_code UNIQUE (edge_node_id, device_code)
    WHERE deleted = 0;
```

> 使用部分唯一索引（WHERE deleted = 0），归档设备不参与唯一性校验，允许同 code 重新上线。

#### SQL 增量文件落位

按项目规范新增至 `sql/postgresql/<yyyy-MM-dd>/` 目录，文件名遵循时间戳命名。

### 3.6 关键流程

#### 3.6.1 device.report 消费主流程

```
边缘端发布 {topicRoot}/telemetry/device-report
    │  Payload 参考：docs/test-data/edge-node-mqtt/03-设备快照上报.json
    │
    ▼
EMQX HTTP Sink → POST /edge/runtime/ingest
    │
    ▼
EdgeRuntimeServiceImpl#ingest()
    │  ├─ 校验 signature（共享密钥鉴权）
    │  ├─ 校验 credential（ACTIVE 凭证）
    │  ├─ 校验 hardware_fingerprint（硬件绑定）
    │  ├─ resolveRuntimeEventType() → "device.report"
    │  ├─ validateRuntimeEventAccess() → 通过
    │  ├─ validateRuntimeEventTopic() → topic 匹配校验
    │  ├─ buildDedupeKey() → 幂等检查
    │  └─ 落原始事件 → EdgeRuntimeEventDO（payloadJson 只存摘要）
    │
    ▼
updateSnapshot(nodeId, event, "device.report", payloadJson, observedAt)
    │  └─ switch 新增 case "device.report"：
    │     ├─ 更新 snapshot.lastDeviceReportAt = observedAt
    │     ├─ 更新 snapshot.deviceCount = devices[].length
    │     └─ （可选）snapshot.onlineDeviceCount = 在线设备数
    │
    ▼
【新增】syncDevicesFromReport(nodeId, payloadJson, observedAt)
    │
    ├─ 1. 解析 payload.devices[] 数组
    │     └─ 每个 device 至少包含：device_code, device_name, device_type, status
    │        可选包含：stream_status, brightness, last_frame_at 等动态属性
    │
    ├─ 2. 遍历每个设备，执行 upsert：
    │     │
    │     ├─ 以 (edgeNodeId, deviceCode) 查找已有记录
    │     │
    │     ├─ 【命中】→ 更新运行态字段：
    │     │   ├─ runtime_status = device.status
    │     │   ├─ stream_status = device.stream_status（如有）
    │     │   ├─ runtime_attrs = 合并其他动态字段到 JSONB
    │     │   ├─ reported_at = observedAt
    │     │   ├─ last_heartbeat_time = observedAt
    │     │   └─ 清除 offline_since（设备重新出现）
    │     │
    │     ├─ 【未命中且非全量上报】→ 新建 ACTIVE 记录：
    │     │   ├─ device_code, name, type, edge_node_id 从 payload 取
    │     │   ├─ source = 'EDGE_REPORT'
    │     │   ├─ runtime_status / stream_status / runtime_attrs 同上
    │     │   └─ reported_at = observedAt
    │     │
    │     └─ 【跨节点冲突】→ (edgeNodeId, deviceCode) 唯一约束违反：
    │        └─ 记录告警日志，跳过该设备，不中断整批处理
    │
    ├─ 3. 全量上报时做"消失检测"：
    │     ├─ 查询 nodeId 下所有未归档设备列表
    │     ├─ 与本次 devices[] 做 diff
    │     └─ 库中有但本次没有的设备：
    │        ├─ 若 offline_since 为空 → 设置 offline_since = observedAt
    │        ├─ 若 offline_since 已超过阈值（如 24h）→ 自动归档
    │        └─ 否则 → 保持 OFFLINE，等待下次确认
    │
    └─ 4. 事务内批量提交
```

#### 3.6.2 边缘端上报节奏（协议约定）

```
┌─────────────────────────────────────────────────┐
│              边缘端 Agent 上报节奏                 │
├─────────────────────────────────────────────────┤
│                                                 │
│  本地设备发现循环（1-5s 检测周期）                 │
│    │                                            │
│    ├─ 检测到新设备 → 立即发增量 device.report     │
│    │   devices[] 仅含新增的设备                   │
│    │                                             │
│    ├─ 检测到设备离线 → 立即发增量 device.report   │
│    │   devices[] 仅含离线设备，status=offline     │
│    │                                             │
│    └─ 检测到属性变化 → 立即发增量 device.report   │
│        devices[] 仅含变化的设备                   │
│                                                 │
│  ════════════════════════════════════════════   │
│                                                 │
│  全量快照定时器（300s 间隔）                      │
│    │                                            │
│    └─ 发全量 device.report                       │
│        devices[] 含所有当前在线设备               │
│        payload 增加 "report_mode": "full_snapshot" │
│                                                 │
└─────────────────────────────────────────────────┘
```

云端通过 `report_mode` 或 `devices[] 是否包含全部已知设备` 来判断是否执行"消失检测"。简化方案：**始终执行消失检测**，依赖 offline_since 阈值防止误判。

#### 3.6.3 设备归档与恢复流程

```
云端归档（手动或自动）
    │
    ├─ 手动：DELETE /edge/device/delete?id={id}
    │   └─ 可选扩展：POST /edge/device/batch-archive { ids: [...] }
    │
    └─ 自动：scanDeviceTimeouts 定时任务（建议 5min 间隔）
        │  遍历 offline_since > 24h 的设备
        └─ 执行逻辑归档
    │
    ▼
更新 edge_device：
    deleted = 1
    archived_at = now()
    status = OFFLINE（归档前终态）
    │
    ▼
关联影响：
    ├─ 设备从默认分页查询中消失（WHERE deleted = 0 过滤）
    ├─ 分组设备数量自动减少
    ├─ 监控告警关闭该设备的活跃告警
    └─ 任务日志中的设备引用不受影响（软删除）

恢复设备
    │
    ▼
POST /edge/device/restore { id: xxx }
    │
    ▼
更新 edge_device：
    deleted = 0
    archived_at = NULL
    status = 根据 offline_since 判断：
              ├─ offline_since 为空 → ACTIVE
              └─ offline_since 非空 → OFFLINE（等待下次上报确认）
```

### 3.7 EdgeRuntimeServiceImpl 变更位置

现有 switch 位于 [EdgeRuntimeServiceImpl.java#L371-L410](../../../moon_cloud_backend/vmesh-module-edge/src/main/java/com/wf/vmesh/module/edge/service/runtime/EdgeRuntimeServiceImpl.java#L371-L410)，需新增 case 分支：

```java
case "device.report" -> {
    snapshot.setLastDeviceReportAt(defaultNow(observedAt));
    // 设备数量在 syncDevicesFromReport 中统计后回填
    snapshot.setDeviceReportStatus("RECEIVED");
}
```

## 6. 风险与回滚

### 风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 高频上报导致 DB 写入瓶颈 | ingest 延迟升高、连接池耗尽 | 批量 upsert + 短窗口去重 + 5min 全量间隔 |
| devices[] payload 过大导致事件表膨胀 | 存储成本上升、查询变慢 | 事件表只存摘要，完整数据落入 edge_device 表 |
| 边缘端全量/增量模式不一致导致消失检测误判 | 正常设备被错误标记离线 | 依赖 offline_since 阈值（如 24h）缓冲，不立即归档 |
| 联合唯一约束导致旧归档设备无法重新上线 | 同 code 设备无法回归 | 使用部分唯一索引 WHERE deleted=0，归档设备不参与约束 |
| 现有前端页面依赖 create 接口 | 页面功能异常 | 前端同步移除新建入口，改为"等待边缘端上报"提示 |

### 回滚方案

- **代码回滚**：移除 switch 中 `case "device.report"` 分支和 `syncDevicesFromReport()` 调用，设备上报回到 default 空分支（丢弃）
- **SQL 回滚**：新增字段可保留（加列不影响既有查询），唯一索引可 DROP
- **接口回滚**：restore/batch-archive 接口不发布即可；create 接口恢复原逻辑；delete 接口恢复物理删除
- **前端回滚**：恢复原有 CRUD 按钮和调用方式
