# 设计方案：边缘节点状态语义与流转

## 1. 目标

统一边缘节点状态语义，明确以下规则：

- 只有 `enabledStatus=1` 的节点允许进入激活流程。
- 只有 `activationStatus=1`（已激活）的节点才允许接收并更新运行态。
- `runtimeStatus` 不再复用 `activationStatus` 枚举，运行态由 runtime 快照独立维护。

## 2. 状态职责分层

### 2.1 `enabledStatus`（台账开关）

- 存储：`edge_node.enabled_status`
- 语义：节点是否允许接入链路。
- 取值：`0=禁用`，`1=启用`
- 作用边界：仅作为准入开关，不表示连接在线或运行健康。

### 2.2 `activationStatus`（激活结果）

- 存储：`edge_node.activation_status`
- 语义：节点当前有效凭证版本是否完成激活闭环。
- 取值：`0=待激活`，`1=已激活`
- 作用边界：只表达“当前凭证版本是否激活成功”，不表达在线/离线/运行中/异常；刷新凭证会重置为待激活。

### 2.3 `registrationStatus`（注册链路阶段）

- 存储：`edge_node_runtime_snapshot.registration_status`
- 当前值：`PREPARED / HANDSHAKING / ACTIVATED`
- 语义：注册流程的阶段进度，用于监控注册链路，不替代激活结果字段。当前 runtime 实现中 `register` 与 `client.connected` 会推进到 `ACTIVATED`，`HANDSHAKING` 保留为链路中间态口径；不再使用旧文档里的 `REGISTERED`。

### 2.3.1 `hardwareFingerprint`（物理节点绑定）

- 存储：`edge_node_credential.hardware_fingerprint`
- 语义：当前 ACTIVE 凭证首次 `register` 时绑定的物理节点指纹。
- 准入约束：`register` 必须提供硬件指纹；同一硬件指纹不能绑定到其他 ACTIVE 凭证；后续 `heartbeat/device.report/task_progress` 必须继续提供相同指纹。
- 安全边界：该机制防止普通凭证复制误用；若攻击者同时复制凭证和可伪造的指纹，仍需更强的设备证书/TPM/安全芯片方案。

### 2.4 `connectionState`（连接态）

- 存储：`edge_node_runtime_snapshot.connection_state`
- 建议值：`UNKNOWN / ONLINE / OFFLINE`
- 语义：MQTT 连接是否在线，由 connect/disconnect 事件驱动。

### 2.5 `runtimeStatus`（运行态）

- 存储：`edge_node_runtime_snapshot.runtime_status`
- 建议值：`OFFLINE / ONLINE / RUNNING / WARNING`
- 语义：节点运行健康态，由 heartbeat/task_progress 等 runtime 事件驱动。
- 前置约束：仅 `activationStatus=1` 时允许更新该字段。

## 3. 流转规则

### 3.1 启用与激活

1. 节点创建后默认 `enabledStatus=1, activationStatus=0`。
2. 若 `enabledStatus=0`，认证与 ingest 直接拒绝，节点保持待激活。
3. 节点完成 `authenticate -> register` 后，更新 `activationStatus=1`。
4. 刷新凭证后旧凭证立即失效，`activationStatus` 重置为 `0`，`connectionState/runtimeStatus` 收敛为离线，等待新凭证重新激活。
5. `register` 首次激活会绑定 `hardwareFingerprint`；绑定后同一凭证只接受相同物理指纹的运行态上报。

### 3.2 连接与运行

1. `client.connected` 更新 `connectionState=ONLINE`，并补齐最近连接时间；若节点已激活，快照会保持可被心跳巡检识别的在线上下文。
2. `client.disconnected` 更新 `connectionState=OFFLINE`，并回落 `runtimeStatus=OFFLINE`。
3. `heartbeat/task_progress` 事件只有在 `activationStatus=1` 时才推进 `runtimeStatus`。
4. 单独发送 heartbeat（未经过激活）不得把节点从待激活推进到已激活。
5. 心跳超时由 `EdgeRuntimeServiceImpl#scanHeartbeatTimeouts` 定时巡检，默认 `vmesh.edge.runtime.heartbeat-scan-interval-ms=30000` 毫秒。
6. 超时判定阈值为 `max(vmesh.edge.runtime.heartbeat-timeout-sec, heartbeatIntervalSec * 3)`；参考时间优先 `lastHeartbeatAt`，其次 `lastRegisterAt`，再其次 `lastConnectedAt`，因此“注册或连接后没有后续心跳”的节点也会被降级为离线。
7. 巡检离线会写入 `connectionState=OFFLINE`、`runtimeStatus=OFFLINE`、`lastDisconnectedAt=now`、`activeTaskKey=null`，并打开 `OFFLINE` 告警；后续心跳恢复时关闭该告警。

## 4. API 映射

## 4.1 节点详情/列表（`/edge/node/get`、`/edge/node/page`）

- 输出字段：`enabledStatus`、`activationStatus`、`registrationStatus`、`connectionState`、`runtimeStatus`。
- 口径约束：移除旧的 `activationStatus -> pending/online/offline/running/warning` 映射。

### 4.2 节点筛选参数（`/edge/node/page`）

- 保留：`enabledStatus`、`activationStatus`
- 新增/明确：`registrationStatus`、`connectionState`、`runtimeStatus`

### 4.3 节点统计（`/edge/node/stats`）

- `pending`：`activationStatus=0`
- `online`：`activationStatus=1 AND connectionState=ONLINE`，并受心跳超时巡检后的快照影响。
- `offline`：`activationStatus=1 AND connectionState=OFFLINE`，包括 `client.disconnected` 和心跳超时巡检降级。
- `running`：`activationStatus=1 AND runtimeStatus=RUNNING`
- `warning`：`activationStatus=1 AND runtimeStatus=WARNING`
- `total`：总节点数（可按租户与删除标识过滤）

### 4.4 runtime ingest 错误口径（`/edge/runtime/ingest`）

- 未启用：拒绝并返回“节点未启用”错误。
- 未激活：拒绝 runtime 状态推进并返回“节点未激活”错误。
- 凭证上下文不一致：`clientId/username/deviceId` 与 ACTIVE 凭证不一致时拒绝。
- 设备绑定不一致：`hardwareFingerprint` 缺失、冲突或不匹配时拒绝。
- 事件字段非法：返回参数错误（事件类型、deviceId、payload 结构）。

## 5. SQL 映射

### 5.1 `edge_node`

- `enabled_status`：准入开关。
- `activation_status`：当前凭证版本的二值激活结果（0/1）。

### 5.1.1 `edge_node_credential`

- `hardware_fingerprint`：当前凭证版本绑定的节点硬件指纹。
- `fingerprint_bound_at`：硬件指纹绑定时间。

### 5.2 `edge_node_runtime_snapshot`

- `registration_status`：注册阶段。
- `connection_state`：连接态。
- `runtime_status`：运行态。

### 5.3 历史数据归一策略

- 旧数据 `activation_status IN (2,3,4)` 统一归一为 `1`（已激活）。
- 运行层语义全部由 runtime snapshot 字段承接。
