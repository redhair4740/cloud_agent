# 边缘数据同步设计方案

## 1. 背景

- 需求来源：`moon_cloud_backend/vmesh-module-edge`、`moon_cloud_frontend/src/views/edge` 需要补一套独立于现有“运行时上报”和“任务下发”的云边数据同步能力。
- 当前现状：后端现有运行时入口是 `POST /edge/runtime/ingest`，负责上行运行时事件入站；现有下行入口是 `POST /edge/task/dispatch`，负责命令/任务型发布；前端现有页面是 `edge/management`、`edge/monitor`、`edge/task` 三块。
- 触发原因：本次需求新增三块能力 `实时同步`、`断网续传`、`一致性校验`，且用户要求和现有数据上报、任务下发接口明确区分。

## 2. 目标

| 编号 | 目标 | 验收指标 |
|------|------|----------|
| G-1 | 新增独立数据同步子域 | 文档层面明确 `sync` 不复用 `runtime` 与 `task` 的路径和语义 |
| G-2 | 覆盖三类能力 | 方案完整描述实时同步、断网续传、一致性校验的后端状态机与前端页面 |
| G-3 | 支持按对象定义单向/双向 | 不同业务对象可配置 `云到边`、`边到云`、`双向待确认` 三种方向 |
| G-4 | 双向冲突必须人工确认 | 文档中禁止自动 merge 双向冲突，必须进入人工确认流 |
| G-5 | 方案能直接指导实现 | 给出模块落点、表设计、接口清单、页面结构、权限/菜单与未验证项 |

## 3. 方案

### 3.1 总体方案

```text
Cloud Admin UI
  -> /admin-api/edge/sync/**
     -> Sync Service
        -> Sync Batch / Sync Item / Sync Snapshot / Sync Conflict
        -> EMQX Publish Service

Edge Node
  -> EMQX Broker
     -> Rule Engine / HTTP Sink
        -> /admin-api/edge/sync/ingest

Cloud Existing Runtime
  -> /admin-api/edge/runtime/authenticate
  -> /admin-api/edge/runtime/ingest

Cloud Existing Task
  -> /admin-api/edge/task/dispatch
```

架构决策：

1. `runtime` 继续负责运行态事件入站，不承载配置同步、补传批次、一致性比对。
2. `task` 继续负责命令/任务型下发，不承载对象级同步批次、差异检测、冲突确认。
3. 新增独立 `sync` 子域，统一承载数据同步的对象模型、批次状态机、差异快照和人工确认。
4. 实时同步与断网续传共用同一批次/明细模型，只是状态推进不同。
5. 双向同步只允许“检测 + 待确认 + 人工裁决”，V1 不做自动双向合并。

### 3.2 领域边界

| 领域 | 后端落点 | 责任 |
|------|----------|------|
| 运行时上报 | `controller/admin/runtime`、`service/runtime` | EMQX 鉴权、运行态入站、快照、监控、任务进度 |
| 任务下发 | `controller/admin/task`、`service/task` | 命令类任务创建、排队、发布、日志 |
| 数据同步 | `controller/admin/sync`、`service/sync`、`dal/dataobject/sync` | 同步对象建模、批次/明细、补传、差异快照、冲突确认 |

边界说明：

- [EdgeRuntimeController#ingest](/mnt/e/projects/AI/moon/moon_cloud_backend/vmesh-module-edge/src/main/java/com/wf/vmesh/module/edge/controller/admin/runtime/EdgeRuntimeController.java:44) 当前是免登录运行态事件入口，入参 `EdgeRuntimeIngestReqVO` 只描述 `eventType/topic/payload`，不适合作为数据同步对象的批量契约。
- [EdgeTaskController#dispatchTask](/mnt/e/projects/AI/moon/moon_cloud_backend/vmesh-module-edge/src/main/java/com/wf/vmesh/module/edge/controller/admin/task/EdgeTaskController.java:39) 当前是单次任务/命令型下发，入参 `EdgeTaskDispatchReqVO` 只描述 `nodeId/taskType/resourceId/dispatchPolicy`，不适合作为多对象同步批次契约。
- 当前 [EdgeTaskServiceImpl#dispatchTask](/mnt/e/projects/AI/moon/moon_cloud_backend/vmesh-module-edge/src/main/java/com/wf/vmesh/module/edge/service/task/EdgeTaskServiceImpl.java:85) 仅支持“离线排队后等待后续机制推进”，并没有现成的“节点恢复上线后自动补传同步对象”能力，因此 V1 同步需要单独补这条机制。

### 3.3 同步对象与方向矩阵

| 对象类型 | 默认方向 | 云端真相源 | 边端真相源 | 冲突策略 |
|----------|----------|------------|------------|----------|
| `NODE_BASE_CONFIG` | 云到边 | 节点主数据、TopicRoot、心跳/超时参数 | 无 | 云端覆盖 |
| `RESOURCE_BINDING` | 云到边 | 资源版本、算法包、配置模板绑定 | 无 | 云端覆盖 |
| `NODE_RUNTIME_FACT` | 边到云 | 无 | 节点运行指标、连接状态、最近事件 | 边端覆盖 |

规则：

1. 单向对象不做反向覆盖。
2. 节点级配置对象如果收到 `sync.conflict`，默认按云端基线自动重发，不再允许“采纳边端回写”。
3. 同一对象类型的方向是稳定契约，不允许前端每次操作临时切换为任意模式。

同步入站说明：

1. `POST /edge/sync/ingest` 是独立于 `POST /edge/runtime/ingest` 的新入口。
2. 该入口接收的是“同步批次/对象级结果”，不是运行时 `heartbeat/device.report/task.progress` 事件。
3. 该入口仍建议沿用 `X-EMQX-SIGNATURE` 共享密钥模式，与现有运行态入口保持安全边界一致，但 VO 独立定义，不复用 `EdgeRuntimeIngestReqVO`。

### 3.6 关键流程

#### 3.6.1 实时同步

1. 管理端在“边缘数据同步”页选择节点和对象范围，调用 `POST /edge/sync/node/{nodeId}/publish`。
2. 后端按对象级拆分生成 `edge_sync_batch` 与若干 `edge_sync_item`，每条明细都绑定 `objectType + objectKey + expectedVersion + expectedHash`。
3. 共享发布服务调用 EMQX `/api/v5/publish`，向 `{topicRoot}/sync/desired/{objectType}` 发布对象数据；payload 至少包含 `object_key`、`version`、`hash`、`content`，而不是只发空壳元数据。
4. 边端处理后通过 `POST /edge/sync/ingest` 回传 `ACK/APPLIED`。
5. 后端推进明细状态 `PENDING -> PUBLISHED -> ACKED -> APPLIED`，并在 `APPLIED` 后把对应 snapshot 收敛到 `IN_SYNC`。

#### 3.6.2 断网续传

1. 后端发起同步时先检查节点连接态；现有连接态真相源仍是运行时快照与节点状态聚合。
2. 若节点离线，则明细进入 `RETRY_WAITING`，不标记为成功。
3. 新增定时任务扫描 `RETRY_WAITING` 明细；节点恢复在线后自动重发，并继续复用原始对象 payload，不重新退化成空壳消息。
4. 每次重发都累计 `retry_count`，超过阈值进入 `FAILED`。
5. 前端允许对 `FAILED` 或 `RETRY_WAITING` 明细手动执行“重试补传”。

#### 3.6.3 一致性校验

1. 管理端调用 `POST /edge/sync/node/{nodeId}/reconcile` 发起对账。
2. 后端发布 compare 请求，边端回传对象 `version/hash` 或字段快照。
3. 后端把云端期望值与边端实际值写入 `edge_sync_snapshot`。
4. 若是节点级配置对象，则直接标记 `IN_SYNC` 或 `DRIFTED`，并允许一键重同步。
5. 节点级配置对象收到 `sync.conflict` 时，后端保留 diff 快照并按云端基线自动重发，不再创建“采纳边端回写”流程。
6. `/edge/sync/conflict/**` 只保留云端确认入口，用于兼容历史遗留冲突记录或人工重发场景。

### 3.7 备选方案与取舍

| 方案 | 结论 | 取舍 |
|------|------|------|
| 继续复用 `/edge/runtime/ingest` | 不选 | 运行时 VO 和事件语义过于宽泛，会把同步对象、运行时遥测、任务进度混在一起 |
| 继续复用 `/edge/task/dispatch` | 不选 | 任务中心是命令型模型，批次/对象/快照/冲突并不是它的职责 |
| 继续把设备对象纳入 sync | 不选 | 设备端只是展示和本地执行载体，云端控制边缘节点，不直接控制设备对象 |
| 保留“采纳边端回写” | 不选 | 节点级同步应以云端为权威，允许边端回写会重新引入设备控制语义 |

## 6. 风险与回滚

- 风险：当前仓内没有边侧代码，本方案只能定义边侧协议和必需行为，不能在本仓直接验证边侧 outbox 持久化。
- 风险：现有 [EdgeRuntimeServiceImpl#authenticate](/mnt/e/projects/AI/moon/moon_cloud_backend/vmesh-module-edge/src/main/java/com/wf/vmesh/module/edge/service/runtime/EdgeRuntimeServiceImpl.java:140) ACL 尚未包含 `sync/#`，后续如果遗漏 ACL 扩展，同步消息会被 broker 拒绝。
- 风险：当前菜单 SQL 只有 `management`、`monitor`、`task` 三个二级菜单，新增 `sync` 页面需要同步 PostgreSQL 菜单与权限种子。
- 回滚方案：若同步功能推进中止，可只保留现有 `runtime` 与 `task` 体系；文档层面移除 `docs/design/边缘数据同步/设计方案.md` 并回退索引引用。
- 未验证项：未读取实时 `/v3/api-docs`，未执行后端编译，未执行前端构建，未执行真实 EMQX/断网恢复/冲突确认联调。
