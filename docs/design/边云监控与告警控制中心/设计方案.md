# 边云监控与告警控制中心设计方案

## 1. 背景

- 目标对象已经明确为 `edge` 业务域，不以 `infra` 为方案主轴；监控、告警、策略、控制都围绕边缘节点、设备与云端协同展开。
- 当前仓库已经有可复用的运行态、监控和任务能力：
  - 后端监控入口：`moon_cloud_backend/vmesh-module-edge/src/main/java/com/wf/vmesh/module/edge/controller/admin/monitor/EdgeMonitorController.java#EdgeMonitorController`，其中 `getOverview`、`getHeartbeatPage`、`getDeviceAbnormalPage`、`getSystemStatusPage`、`getAlertPage` 已存在。
  - 监控聚合服务：`moon_cloud_backend/vmesh-module-edge/src/main/java/com/wf/vmesh/module/edge/service/monitor/EdgeMonitorServiceImpl.java#getOverview`、`getAlertPage` 等方法已存在。
  - 运行态入站与任务链路：`moon_cloud_backend/vmesh-module-edge/src/main/java/com/wf/vmesh/module/edge/controller/admin/runtime/EdgeRuntimeController.java`、`moon_cloud_backend/vmesh-module-edge/src/main/java/com/wf/vmesh/module/edge/controller/admin/task/EdgeTaskController.java` 已存在。
  - 前端现有页面：`moon_cloud_frontend/src/views/edge/monitor/index.vue`、`moon_cloud_frontend/src/views/edge/task/index.vue`、`moon_cloud_frontend/src/views/edge/management/index.vue`。
- 当前缺口不是再做一个基础设施首页，而是把“实时告警、告警历史、告警策略、硬件控制”整合成一个云边联合的业务中心。

## 2. 目标

| 编号 | 目标 | 说明 |
|------|------|------|
| G-1 | 建立独立业务中心 | 形成“运行总览 + 告警 + 策略 + 控制”统一入口 |
| G-2 | 统一云边告警口径 | 同时承接边缘端上报告警和云端派生告警 |
| G-3 | 建立告警到控制闭环 | 支持人工处置、自动联动、执行结果回写 |
| G-4 | 最大化复用现有能力 | 复用 `runtime`、`monitor`、`task` 真相源，不新建第二套执行状态机 |

## 3. 最新方案

### 3.1 总体定位

- 本中心定位为 `edge` 域下的业务工作台，不是项目级 `infra` 监控首页。
- V1 聚焦五个一级能力：`运行总览`、`实时告警`、`告警历史`、`告警策略`、`硬件控制`。
- 云端告警与边缘端告警采用“双来源、单模型”统一收口。
- 硬件控制复用现有 `edge_task`、`edge_task_log` 链路，不平行建设第二套控制任务系统。

### 3.2 页面结构

| 页面 | 核心区块 | 主表/主内容 | 主要动作 | 详情容器 |
|------|----------|-------------|----------|----------|
| 运行总览 | 指标卡、重点异常、趋势分布、异常节点/设备列表、快捷入口 | 在线节点、在线设备、异常设备、活跃告警、失败任务、控制失败 | 查看实时告警、查看控制记录、查看节点详情、查看设备详情 | 节点详情抽屉、设备详情抽屉 |
| 实时告警 | 筛选区、统计条、实时告警表 | 当前 `OPEN / ACKED / SILENCED` 告警 | 详情、确认、静默、关闭、一键控制 | 告警详情抽屉、控制弹窗 |
| 告警历史 | 筛选区、历史表、生命周期时间线 | 触发、重复触发、恢复、关闭、联动控制结果 | 查看时间线、查看控制结果 | 时间线抽屉、控制结果抽屉 |
| 告警策略 | 筛选区、策略表、策略表单 | 作用范围、触发条件、恢复条件、动作配置、优先级、启停状态 | 新增、编辑、启停、复制 | 策略编辑抽屉 |
| 硬件控制 | 筛选区、控制记录表、快捷控制区 | 人工控制、自动控制、控制任务状态、失败原因 | 发起控制、重试、查看任务日志 | 控制下发弹窗、任务日志抽屉 |

页面设计原则只保留三条：

1. 保持后台管理页风格，沿用现有 `src/views/edge/*` 的表单、表格、抽屉工作流。
2. 首页负责发现异常，实时告警负责处置当前问题，历史页负责审计，策略页负责治理，控制页负责执行。
3. 页面之间以“异常发现 -> 告警处置 -> 控制执行 -> 结果回看”串成闭环，不再散落在多个独立页面中。

### 3.3 云边联合告警模型

| 来源类型 | 说明 | 典型场景 |
|----------|------|----------|
| `EDGE_REPORTED` | 由边缘端或设备端发现异常后主动上报 | 摄像头断流、外设故障、本地进程异常、设备温度异常 |
| `CLOUD_EVALUATED` | 由云端基于运行态、心跳、任务和历史数据派生 | 节点离线、心跳超时、任务失败、异常持续未恢复 |

统一告警实例至少保留以下字段：

- `sourceType`
- `alertType`
- `severity`
- `status`
- `nodeId`
- `deviceId`
- `taskKey`
- `strategyId`
- `firstTriggeredAt`
- `lastTriggeredAt`
- `recoveredAt`
- `summary`
- `detailJson`
- `upstreamEventId`

统一口径要求：

1. 同一主体、同一策略、同一告警类型的未结束告警应合并为一个当前态实例，只更新最近触发时间和明细。
2. 历史变化不覆盖当前态，而是单独沉淀为生命周期事件。
3. 边缘端上报与云端派生只在来源和明细上不同，不再建设两套页面和两套状态机。

### 3.4 告警与控制闭环

核心流程固定为：

1. `runtime ingest` 接收边缘事实或系统事件。
2. `alert evaluate` 生成或更新告警实例，并写入告警历史事件。
3. 命中策略后决定仅告警、通知、自动控制或创建任务。
4. 需要执行控制时，由 `control` 编排后复用 `task dispatch` 下发到边缘端。
5. 边缘端回传执行状态后，云端更新 `edge_task`、`edge_task_log`，并回写控制记录与告警历史。

关键约束：

- 控制执行链路继续以 `edge_task` 为真相源。
- 告警历史必须记录控制触发、控制成功、控制失败等事件，否则历史页无法还原处置链路。
- 页面层不直接拼接任务载荷，由 `control` 子域负责把页面动作转换为标准控制命令。

### 3.5 模块边界与微服务演进

当前阶段建议：

- 后端继续落在 `moon_cloud_backend/vmesh-module-edge`。
- 前端继续落在 `moon_cloud_frontend/src/views/edge` 和 `moon_cloud_frontend/src/api/edge`。
- 在 `edge` 域内按五个逻辑子域组织：`runtime`、`monitor`、`alert`、`control`、`task`。

当前不建议把监控整体从 `edge` 物理拆出，原因只有两点：

1. 运行态入站、告警生成、控制编排、任务回执目前都强依赖同一批 `edge` 真相源。
2. 现在拆服务会提前引入跨服务一致性和补偿复杂度，但业务边界还在收敛期。

未来如果需要微服务化，建议拆分为：

```text
edge-core-service
  - node
  - device
  - runtime
  - task

edge-alert-control-service
  - realtime alert
  - alert history
  - alert strategy
  - control orchestration
```

结论是：未来也不建议单独拆“monitor 服务”，更合理的拆分对象是 `alert + control`。

### 3.6 数据模型要点

| 类型 | 表/对象 | 处理建议 | 作用 |
|------|---------|----------|------|
| 复用 | `edge_node_runtime_snapshot` | 继续复用 | 运行总览、云端派生告警判断输入 |
| 复用 | `edge_node_metric_minute` | 继续复用 | 趋势图、阈值判断、历史聚合 |
| 扩展 | `edge_monitor_alert` | 继续作为当前态告警实例表，补充来源、策略、恢复、静默等字段 | 实时告警主表 |
| 新增 | `edge_alert_history` | 新增生命周期事件表 | 触发、恢复、关闭、控制结果时间线 |
| 新增 | `edge_alert_strategy` | 新增策略主表 | 触发条件、恢复条件、动作配置 |
| 新增 | `edge_alert_strategy_scope` | 新增作用域明细表 | 节点/设备/分组等作用范围 |
| 新增 | `edge_control_record` | 新增轻量控制记录表，通过 `task_key` 关联任务体系 | 控制业务视角与告警联动追溯 |
| 复用 | `edge_task`、`edge_task_log` | 继续复用 | 控制执行、回执、失败原因 |

数据建模约束：

1. 当前态和历史态分离：`edge_monitor_alert` 负责当前态，`edge_alert_history` 负责时间线。
2. 控制请求和控制执行分离：`edge_control_record` 负责业务记录，`edge_task` 负责执行状态。
3. 易变规则和异构明细优先放 `jsonb`，如 `detailJson`、`triggerRuleJson`、`recoverRuleJson`、`actionConfigJson`、`receiptJson`。
4. 当前不建议新增首页快照表、独立控制任务表、通知记录表，避免过早设计。

## 4. 影响范围

- 后端：`moon_cloud_backend/vmesh-module-edge/**`
- 前端：`moon_cloud_frontend/src/views/edge/**`、`moon_cloud_frontend/src/api/edge/**`
- 数据库：扩展 `edge_monitor_alert`，新增告警历史、策略、策略作用域、控制记录相关表，复用 `edge_task`、`edge_task_log`
- 文档：相关接口说明继续落在 [API文档](../../api/边云监控与告警控制中心/API文档.md)

## 5. 验收标准

- [ ] 管理端形成 5 个核心页面：运行总览、实时告警、告警历史、告警策略、硬件控制。
- [ ] 实时告警和历史告警同时支持 `EDGE_REPORTED`、`CLOUD_EVALUATED` 两类来源。
- [ ] 告警支持人工处置和自动联动控制，且控制结果能回写到历史时间线。
- [ ] 控制链路复用 `edge_task`、`edge_task_log`，不存在第二套并行执行状态机。
- [ ] 后续实现阶段至少补齐后端编译验证、前端类型检查和云边联调验证。

## 6. 风险与未验证项

- 风险：如果不新增 `edge_alert_history`，历史页只能看到实例快照，无法还原处置过程。
- 风险：如果边缘端告警 payload 不统一，云端难以稳定合并 `EDGE_REPORTED` 与 `CLOUD_EVALUATED`。
- 风险：如果控制页直接拼任务载荷，会把页面耦合到执行协议，后续协议变更成本高。
- 未验证：本轮只做文档收敛，未读取实时 OpenAPI，未生成 PostgreSQL 增量 SQL，未执行编译、测试、前端构建和真实云边联调。
