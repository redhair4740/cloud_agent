# 边云监控与告警控制中心设计方案

## 1. 背景

- 需求来源：用户希望围绕边缘节点、设备端与云平台自身的运行协同，建设实时告警、告警历史、告警策略、硬件控制等目标能力，并明确“云端告警需要结合边缘端告警”。
- 当前现状：项目已存在 `moon_cloud_backend/vmesh-module-edge` 的运行时入站、边缘监控、任务下发、数据同步等能力；前端已存在 `moon_cloud_frontend/src/views/edge/monitor/index.vue`、`moon_cloud_frontend/src/views/edge/task/index.vue`、`moon_cloud_frontend/src/views/edge/management/index.vue`。
- 触发原因：当前仓库已有“监控页”和“任务中心”，但还没有形成完整的“告警治理 + 控制闭环”业务中心，尤其缺少云边联合告警口径、告警历史沉淀、策略配置和从告警触发硬件控制的闭环。

## 2. 目标

| 编号 | 目标 | 验收指标 |
|------|------|----------|
| G-1 | 建立面向 Edge 的业务监控中心 | 管理端存在单独业务中心，覆盖运行总览、告警、策略、控制 |
| G-2 | 统一云端与边缘端告警口径 | 实时告警与历史告警能同时展示 `EDGE_REPORTED` 和 `CLOUD_EVALUATED` 两类来源 |
| G-3 | 建立告警到控制的闭环 | 告警可触发人工或自动控制，并回写控制结果 |
| G-4 | 最大化复用现有 edge 能力 | 复用 `edge/runtime`、`edge/monitor`、`edge/task` 现有真相源，不以 `infra` 作为主轴 |

## 3. 方案

### 3.1 总体方案

边云监控与告警控制中心定位为 **`edge` 业务域下的统一工作台**，主对象是边缘节点、设备、云端任务与告警策略，不再以 `infra` 监控入口为主轴。

总体结构建议如下：

```text
边云监控与告警控制中心
  ├─ 运行总览
  │   ├─ 在线节点数
  │   ├─ 在线设备数
  │   ├─ 异常设备数
  │   ├─ 活跃告警数
  │   ├─ 失败任务数
  │   └─ 控制失败数
  ├─ 实时告警
  │   ├─ 云端派生告警
  │   ├─ 边缘端上报告警
  │   └─ 告警处置入口
  ├─ 告警历史
  │   ├─ 生命周期记录
  │   ├─ 恢复记录
  │   └─ 联动控制结果
  ├─ 告警策略
  │   ├─ 阈值策略
  │   ├─ 设备异常策略
  │   └─ 联动动作策略
  └─ 硬件控制
      ├─ 人工控制
      ├─ 自动控制
      └─ 控制任务记录
```

核心设计决策：

1. 本中心不是基础设施监控首页，而是 `edge` 业务中心。
2. 边缘端告警和云端告警采用“双来源、单模型”收口。
3. 硬件控制不单独建设第二套任务系统，而是优先复用现有 `edge/task` 下发链路。
4. V1 聚焦边缘节点、设备、任务、告警与控制闭环，`infra` 相关监控只保留弱关联，不作为主设计对象。

### 3.2 现状证据与可复用能力

#### Edge 侧已有真相源能力

- 后端监控入口：`/edge/monitor/**`，见 `moon_cloud_backend/vmesh-module-edge/src/main/java/com/wf/vmesh/module/edge/controller/admin/monitor/EdgeMonitorController.java#EdgeMonitorController`
- 监控聚合逻辑：`EdgeMonitorServiceImpl#getOverview`、`getHeartbeatPage`、`getDeviceAbnormalPage`、`getSystemStatusPage`、`getAlertPage`
- 运行态入站：`POST /edge/runtime/ingest`，用于接收上行运行时事件，见 [Edge 边缘模块 API 文档](../../api/Edge边缘模块/API文档.md)
- 任务下发：`POST /edge/task/dispatch`、任务分页、详情、日志、重试、撤回，见 [Edge 边缘模块 API 文档](../../api/Edge边缘模块/API文档.md)
- 数据存储：`edge_node_runtime_snapshot`、`edge_node_metric_minute`、`edge_monitor_alert`、`edge_task`、`edge_task_log`
- 设备上报设计已经明确设备异常需联动监控告警，见 [边缘设备上报设计方案](../边缘设备上报/设计方案.md)

结论：

- 现有仓库已经具备“监控雏形 + 控制雏形”。
- 缺的不是基础设施入口，而是“云边联合告警模型、告警策略和控制闭环”。

### 3.3 页面信息架构

页面建议仍采用后台管理风格，不做大屏化设计，优先和现有 `moon_cloud_frontend/src/views/edge/monitor/index.vue`、`moon_cloud_frontend/src/views/edge/task/index.vue` 的结构保持一致。

页面建议拆为 5 个一级功能页：

#### A. 运行总览

展示最关键的业务运行指标：

- 在线边缘节点数
- 在线设备数
- 异常设备数
- 活动告警数
- 失败任务数
- 控制失败数

总览页重点不是展示所有明细，而是回答三件事：

1. 现在是否有异常在发生
2. 这些异常来自哪些节点/设备
3. 是否已经进入控制或恢复流程

#### B. 实时告警

实时告警页只看“当前还在处理中的告警”，建议包含：

- 告警来源：`EDGE_REPORTED` / `CLOUD_EVALUATED`
- 告警级别
- 节点/设备归属
- 告警类型
- 首次触发时间
- 最近触发时间
- 当前状态
- 推荐动作
- 一键处置/一键控制入口

#### C. 告警历史

告警历史页关注“完整生命周期”，而不是当前态，建议包含：

- 触发记录
- 重复触发记录
- 人工确认记录
- 自动恢复记录
- 人工关闭记录
- 联动控制结果

#### D. 告警策略

告警策略页建议配置以下维度：

- 作用范围：节点 / 设备 / 设备类型 / 分组
- 触发条件：阈值、持续时间、连续次数、时间窗口
- 恢复条件：恢复阈值、稳定持续时间
- 动作类型：仅告警 / 通知 / 自动控制 / 创建任务
- 启停状态与优先级

#### E. 硬件控制

硬件控制页建议同时支持：

- 人工控制：人工下发重启、启停、模式切换、配置刷新等操作
- 自动控制：由告警策略自动触发控制动作
- 控制任务跟踪：展示任务状态、日志、回执、失败原因

### 3.4 云边联合告警模型

云端与边缘端告警采用“双来源、单模型”统一设计。

#### 边缘端/设备端告警

- 由边缘端本地规则或设备端逻辑发现并上报
- 更适合发现贴近硬件真相的异常：
  - 摄像头断流
  - 外设故障
  - 本地温度异常
  - 本地进程异常

#### 云端派生告警

- 由云端根据运行态、任务、心跳、设备上报结果计算得出
- 更适合发现跨时间窗口、跨设备、跨任务的异常：
  - 节点离线
  - 心跳超时
  - 任务失败/超时
  - 长时间未恢复
  - 设备异常持续存在

#### 统一告警模型

建议所有告警统一至少包含：

- `sourceType`：`EDGE_REPORTED` / `CLOUD_EVALUATED`
- `alertType`
- `severity`
- `status`
- `nodeId`
- `deviceId`
- `taskKey`
- `strategyId`
- `firstTriggeredAt`
- `lastTriggeredAt`
- `recoveredAt`
- `summary`
- `detailJson`
- `upstreamEventId`

### 3.4 模块边界与归属

#### 后端归属

- 本中心建议落在 `moon_cloud_backend/vmesh-module-edge`
- 原因：
  - 主体对象是节点、设备、告警、策略、控制任务
  - 现有真相源已经在 `edge` 域内
  - `infra` 未来可能被舍弃，不应成为这次方案的架构锚点

#### 跨模块依赖边界

- 不新增对 `infra` 的强依赖
- 告警、策略、控制应围绕 `edge/runtime`、`edge/monitor`、`edge/task`、`edge/device` 展开
- 若后续需要对接通知、日志或平台级组件，应通过轻量 API/配置接入，而不是让 `edge` 直接依赖 `infra` 页面模型

#### 前端归属

- 页面建议落在 `moon_cloud_frontend/src/views/edge`
- 原因：
  - 本质是边缘业务中心
  - 与现有 `edge/monitor`、`edge/task` 更容易形成统一导航和一致交互
  - 后续可与 `edge/management`、`edge/task` 形成工作台闭环

  - 新增建议接口 `/edge/alert/strategy/**`
  - 新增建议接口 `/edge/control/**`
- 接口状态：当前仅完成静态设计，未生成 OpenAPI、未联调
- 状态确认人：未确认

建议的 V1 接口职责：

#### `GET /edge/alert/realtime/page`

返回当前未恢复、未关闭、未完成处置的告警列表。

#### `GET /edge/alert/history/page`

返回完整告警历史与处置记录。

### 3.6 数据模型变更（建议）

原“统一监控中心”方案里写的是 V1 不新增业务表；在加入“实时告警、告警历史、告警策略、硬件控制”后，这个边界需要调整。

建议至少拆成三层：

#### 1. 告警实例表

- 继续复用/扩展现有 `edge_monitor_alert`
- 作为当前态告警实例，用于实时告警列表

#### 2. 告警历史事件表

- 新增建议表
- 用于记录：触发、确认、恢复、关闭、自动控制、控制结果等生命周期事件

#### 3. 告警策略表

- 新增建议表
- 用于保存触发条件、恢复条件、动作和作用范围

#### 4. 硬件控制任务

- 不建议新建平行“控制任务系统”
- 建议复用现有 `edge_task` 与 `edge_task_log`
- 通过新的 `taskType` / `commandKey` 承载硬件控制语义

### 3.7 关键流程

#### 流程 1：云边联合告警生成

1. 边缘端/设备端发现异常并主动上报，或云端通过运行态与任务态派生告警。
2. 云端统一转换为告警实例。
3. 若同一主体同一策略下已有未结束告警，则更新最近触发时间；否则创建新实例。
4. 同步写入告警历史事件。

#### 流程 2：策略匹配与联动

1. 告警实例生成后，匹配已启用策略。
2. 根据策略决定：
   - 仅告警
   - 通知
   - 自动控制
3. 若需要自动控制，则创建控制任务。

#### 流程 3：硬件控制执行回写

1. 云端通过现有 `edge/task/dispatch` 链路下发控制命令。
2. 边缘端执行后通过 `task-progress` 回传执行状态。
3. 云端更新 `edge_task` / `edge_task_log`。
4. 同时向告警历史写入“控制成功/失败”事件。

### 3.8 备选方案与取舍

#### 方案 A：继续做项目级统一监控首页

- 优点：结构简单
- 缺点：无法承载告警治理和控制闭环，且会把重点带偏到基础设施入口

#### 方案 B：把告警、策略、控制继续散落在 monitor / task / device 多页里

- 优点：改动小
- 缺点：用户操作路径碎片化，无法形成“发现异常 -> 处置 -> 控制 -> 回看结果”的闭环

#### 方案 C：建设 `edge` 域下的边云监控与告警控制中心

- 优点：贴合当前业务对象和未来演进方向，最大化复用现有 `edge` 真相源
- 缺点：需要补告警历史与策略建模，设计量高于单纯首页聚合

结论：V1 选 **方案 C**。

### 3.9 微服务演进路线与拆分边界

当前阶段**不建议**把监控整体从 `edge` 模块物理拆出，建议先在 `moon_cloud_backend/vmesh-module-edge` 内完成逻辑子域收口，再根据真实运行压力和团队边界决定是否拆分微服务。

#### 当前不拆的原因

1. 运行态入站、告警生成、任务下发、控制回执目前都在 `edge` 主链路内，强依赖同一批真相源对象。
2. 告警与控制的页面、字段、策略和生命周期还处于设计收敛期，业务边界尚未稳定。
3. 现在拆分会立即引入跨服务查询、事件一致性、幂等、失败补偿和回写链路复杂度。
4. 当前仓库规则更偏向“先把业务能力收口在所属域内，再通过契约暴露给外部”，不鼓励在业务语义未稳定时先物理拆服务。

#### 当前阶段建议的逻辑子域

即使仍保留在 `vmesh-module-edge` 内，也建议按下面 5 个逻辑子域组织代码与职责：

| 子域 | 当前建议职责 | 备注 |
|------|--------------|------|
| `runtime` | 接收边缘上行事件、维护运行态事实、输出运行态快照 | 不直接承载策略编排 |
| `monitor` | 运行总览、心跳/状态/异常聚合视图 | 偏读模型与聚合视图 |
| `alert` | 告警实例、告警历史、告警策略、云边联合告警收口 | 告警中心核心子域 |
| `control` | 告警联动控制、人工控制编排、控制结果回写 | 不直接替代任务系统 |
| `task` | 命令任务下发、状态推进、执行日志、边端回执 | 继续作为控制动作执行载体 |

#### 子域之间的交互约束

- `alert` 不直接依赖 `runtime`、`device`、`task` 的 Mapper / DO 作为常规实现手段。
- `control` 不直接改写 `runtime` 快照或告警实例状态，而是通过任务执行结果回写。
- `monitor` 只负责聚合展示，不成为告警判断或控制编排入口。
- 跨子域协作优先通过 service 契约、应用服务、DTO/VO 或内部 API 实现。

#### 未来更合理的拆分方向

未来若需要微服务化，**更合理的拆分对象不是“monitor 服务”，而是“alert-control 服务”**。

建议的未来拆分形态：

```text
edge-core-service
  - node
  - device
  - runtime
  - sync
  - task

edge-alert-control-service
  - realtime alert
  - alert history
  - alert strategy
  - control orchestration
```

这样拆的原因：

- `edge-core-service` 保留边缘接入、运行态和任务执行真相源
- `edge-alert-control-service` 独立承载告警治理和控制编排
- 比“把 monitor 整体拆出去”更符合业务语义，也更容易独立演进

#### 未来拆分前的触发条件

建议至少满足以下条件中的大部分，再考虑物理拆分：

1. 告警策略、告警生命周期、控制动作模型已经稳定。
2. 告警与控制流量明显增大，已经需要独立扩容或独立发布。
3. 团队职责开始分离，存在长期维护告警/控制中心的独立 ownership。
4. 当前单模块实现已因发布节奏、性能或容灾要求受到明显限制。

#### 当前阶段的工程建议

- 先在 `vmesh-module-edge` 内落地 `alert` 与 `control` 子包/子域，形成稳定代码边界。
- 尽量把“事实写入”“告警判断”“控制编排”“页面聚合”四类逻辑分开，不再堆进单个 `monitor` service。
- 对未来可能跨服务的交互，优先先收口成内部契约，而不是先收口成共享 DO/Mapper。

### 3.10 模块分层图与包结构建议

为兼容“当前单模块落地”和“未来可拆分微服务”两种形态，建议在 `vmesh-module-edge` 内先把代码按稳定业务子域和职责分层组织，而不是继续按“页面功能临时堆叠”方式扩展。

#### 推荐分层关系

```text
edge/controller/admin
  ├─ runtime      -> 运行态接入与只读排障入口
  ├─ monitor      -> 运行总览与聚合查询
  ├─ alert        -> 实时告警、历史、策略
  ├─ control      -> 人工控制入口
  └─ task         -> 命令任务执行与日志

edge/service
  ├─ runtime      -> 运行态事实维护
  ├─ monitor      -> 聚合读模型
  ├─ alert        -> 告警实例/历史/策略判断
  ├─ control      -> 控制编排
  └─ task         -> 任务执行状态机

edge/dal
  ├─ dataobject/runtime
  ├─ dataobject/monitor
  ├─ dataobject/alert
  ├─ dataobject/control
  ├─ dataobject/task
  ├─ mapper/runtime
  ├─ mapper/monitor
  ├─ mapper/alert
  ├─ mapper/control
  └─ mapper/task
```

#### 后端包结构建议

建议目录结构如下：

```text
moon_cloud_backend/vmesh-module-edge/src/main/java/com/wf/vmesh/module/edge
  ├─ controller/admin/runtime
  ├─ controller/admin/monitor
  ├─ controller/admin/alert
  │   ├─ AlertRealtimeController
  │   ├─ AlertHistoryController
  │   └─ AlertStrategyController
  ├─ controller/admin/control
  │   └─ EdgeControlController
  ├─ service/runtime
```

#### 每层职责边界

1. `runtime`
- 只负责接收边缘端事实、维护快照、触发基础事件。
- 不在这里直接承载复杂策略配置页面逻辑。

2. `monitor`
- 只负责运行总览、状态聚合、读模型输出。
- 不直接写策略，不直接发控制命令。

3. `alert`
- 负责告警实例生成、生命周期流转、历史沉淀、策略匹配。
- 是未来最可能独立微服务化的核心边界。

4. `control`
- 负责把“人工操作”或“策略动作”转换为可执行控制命令。
- 不自己维护底层命令执行状态机，而是委托 `task` 执行。

5. `task`
- 负责命令真正下发、跟踪边缘回执、维护日志和状态推进。
- 保持“执行载体”角色，不吸收告警策略判断。

#### 推荐调用链

建议后端调用链固定为：

```text
runtime ingest
  -> runtime facts
  -> alert evaluate
  -> alert instance/history
  -> control orchestration (optional)
  -> task dispatch (optional)

monitor query
  -> runtime snapshot + alert summary + task summary
  -> aggregate view
```

这个调用链的意义是：

- `runtime` 产事实
- `alert` 做判断
- `control` 做编排
- `task` 做执行
- `monitor` 做展示

避免把“事实、判断、执行、展示”混在一个 service 里。

#### 当前实现阶段的落地建议

- 第一阶段先不改动物理模块，只按包结构新增 `alert`、`control` 子域。
- 现有 `monitor` 中与告警判断强相关的逻辑，后续应迁到 `service/alert`。
- 现有硬件控制若先复用 `task/dispatch`，也应在 `control service` 做一次命令编排适配，不要让页面直接拼任务载荷。
- 若未来拆微服务，优先迁移整个 `alert + control` 子域，而不是逐类 controller 零散迁移。

### 3.13 数据库表结构草案

本节只给出目标态表结构草案和语义边界，不代表已经落库。  
当前项目数据库口径仍以 PostgreSQL 增量 SQL 为准；本节不替代后续正式迁移脚本。

#### 3.13.1 设计原则

1. 能复用现有 `edge` 真相源表的场景不新造平行表。
2. 当前态与历史态分离：
- 当前态用于页面实时查询
- 历史态用于时间线、审计和联动回放
3. 控制执行复用任务链路，不单独复制一套“执行状态机”。
4. 策略配置、告警实例、历史事件三类表分开，避免一张表混杂配置与运行态。

#### 3.13.8 表关系建议

建议关系如下：

```text
edge_node_runtime_snapshot  -> 云端派生告警判断输入
edge_node_metric_minute     -> 趋势/阈值判断输入
edge_monitor_alert          -> 当前态告警实例
edge_alert_history          -> 告警生命周期历史
edge_alert_strategy         -> 告警策略主表
edge_alert_strategy_scope   -> 策略作用域明细
edge_control_record         -> 控制业务记录
edge_task                   -> 控制执行任务
edge_task_log               -> 控制执行日志
```

主要关联键：

| 关系 | 建议键 |
|------|--------|
| 告警实例 -> 告警历史 | `edge_alert_history.alert_id = edge_monitor_alert.id` |
| 告警实例 -> 控制记录 | `edge_control_record.alert_id = edge_monitor_alert.id` |
| 控制记录 -> 任务 | `edge_control_record.task_key = edge_task.task_key` |
| 任务 -> 任务日志 | `edge_task_log.task_key = edge_task.task_key` |
| 策略 -> 作用域 | `edge_alert_strategy_scope.strategy_id = edge_alert_strategy.id` |

#### 3.13.9 建议保留为 JSON 的字段

以下字段更适合先走 `jsonb`，避免 V1 过早过细拆表：

| 字段 | 原因 |
|------|------|
| `edge_monitor_alert.detail_json` | 不同告警类型明细形状差异大 |
| `edge_alert_history.detail_json` | 历史事件 payload 差异大 |
| `edge_alert_strategy.trigger_rule_json` | 规则形状仍会演进 |
| `edge_alert_strategy.recover_rule_json` | 恢复规则仍会演进 |
| `edge_alert_strategy.action_config_json` | 动作配置会随控制类型变化 |
| `edge_control_record.inline_payload_json` | 控制命令参数高度异构 |
| `edge_control_record.receipt_json` | 边缘端回执格式可能分阶段收敛 |

## 5. 验收标准

- [ ] 功能验收：具备运行总览、实时告警、告警历史、告警策略、硬件控制 5 个核心能力
- [ ] 契约验收：边缘端上报告警、云端派生告警、控制任务回执三条链路的字段口径一致
- [ ] 验证要求：后续实现时至少执行后端模块编译、前端类型检查，并核对告警触发、恢复、控制失败、自动联动四类场景

## 6. 风险与回滚

- 风险：若不单独建设告警历史事件模型，历史页只能看到实例快照，无法还原处置链路
- 风险：当前尚无统一原型图，页面布局和交互层级仍可能调整
- 风险：边缘端告警事件格式若不统一，云端无法稳定合并 `EDGE_REPORTED` 与 `CLOUD_EVALUATED`
- 回滚方案：若后续原型收窄，可先只实现“实时告警 + 硬件控制 + 任务联动”，策略页和历史页延后
- 未验证项：本轮未读取实时 OpenAPI，未执行编译验证，未执行前端构建，未核对现有边缘端真实告警 payload 格式，也未验证控制命令枚举是否满足全部硬件控制场景
