# 边缘监控与任务流水原型对齐详细设计

## 1. 背景

- 需求来源：依据最新原型 `wf_vmesh_cloud_design/src/views/EdgeHealthMonitor.vue` 与 `wf_vmesh_cloud_design/src/views/EdgeTaskRecord.vue`，补充并修订既有设计文档 `docs/design/2026-04-22/00-00-00-设计方案-边云协同Edge全链路生产级设计.md`。
- 当前现状：
  - 旧设计的 `边缘监控` 页面包含“注册链路跟踪表、心跳实时列表、高负载节点排行、活跃告警列表”等区域，见旧设计 `docs/design/2026-04-22/00-00-00-设计方案-边云协同Edge全链路生产级设计.md#14.4`。
  - 最新 `EdgeHealthMonitor.vue` 已删除“边缘注册全链路跟踪”，页面主体变为三块：边缘节点心跳监控、边缘节点连接设备异常状态、边缘节点系统状态预警，证据见 `wf_vmesh_cloud_design/src/views/EdgeHealthMonitor.vue:78`、`:111`、`:140`。
  - 最新 `EdgeTaskRecord.vue` 仍围绕任务统计、任务筛选、任务流水表格、日志抽屉组织，证据见 `wf_vmesh_cloud_design/src/views/EdgeTaskRecord.vue:3`、`:41`、`:65`、`:120`。
  - 当前生产前端已有 `WF_VMesh_Coud_UI/src/api/edge/monitor.ts` 与 `WF_VMesh_Coud_UI/src/api/edge/task.ts`，当前后端已有 `EdgeMonitorController` 与 `EdgeTaskController`，但监控侧尚未覆盖“连接设备异常状态”目标契约。
- 触发原因：旧设计仍保留注册链路视角，和最新原型目标不一致；需要形成新的详细设计与 API 接口文档，作为后续前后端实现和契约调整依据。

## 2. 目标

| 编号 | 目标 | 验收指标 |
|------|------|----------|
| G-1 | 边缘监控页面按最新原型收敛为三块 | 页面不再展示注册链路，展示心跳、异常设备、系统状态三类监控信息 |
| G-2 | 正常连接设备不展示 | 设备异常列表只返回 `OFFLINE`、`STREAM_LOST`、`FRAME_DROP`、`AUTH_FAILED` 等异常记录，空态为“暂无异常设备” |
| G-3 | 系统状态预警延续高负载节点预警体验 | 保留排序、表格、进度条表达，字段扩展到 CPU、NPU/GPU、内存、存储、温度、建议 |
| G-4 | 任务流水页面契约化 | 支持今日统计、关键词/类型/状态筛选、分页流水、日志抽屉、撤回、重试 |
| G-5 | 明确新旧契约差异 | 明确哪些接口可复用、哪些字段需新增、哪些旧接口应停止在页面使用 |

## 3. 方案

### 3.1 总体方案

本次是原型驱动的设计修订，不修改现有业务代码。后续落地时按“前端展示目标态 + 后端契约最小补齐”的方式推进：

1. `EdgeHealthMonitor.vue` 对应生产页面 `WF_VMesh_Coud_UI/src/views/edge/monitor/index.vue`。
2. `EdgeTaskRecord.vue` 对应生产页面 `WF_VMesh_Coud_UI/src/views/edge/task/index.vue` 与详情抽屉 `WF_VMesh_Coud_UI/src/views/edge/task/components/TaskDetailDrawer.vue`。
3. 监控页移除注册链路消费，不再调用或展示 `/edge/monitor/registration/page`。
4. 监控页新增设备异常列表契约，后端从边缘节点绑定设备、设备运行态、告警事件中聚合异常设备。
5. 系统状态预警可由旧“高负载节点”能力演进而来，但页面语义改为“系统主要参数监控”，后端应返回更完整的系统状态字段。
6. 任务流水继续复用当前任务中心接口体系，统一使用大写枚举：`OTA`、`ALGORITHM`、`CONFIG`、`REBOOT`；状态使用 `CREATED`、`QUEUED`、`DISPATCHED`、`ACKNOWLEDGED`、`PROCESSING`、`SUCCESS`、`FAILED`、`CANCELED`、`TIMEOUT`。

### 3.4 数据模型变更

#### 3.4.1 监控聚合模型

边缘监控页目标数据不要求新建单体大表，优先从现有运行态表、设备台账表、告警表和任务表聚合：

| 目标数据 | 推荐来源 | 说明 |
|----------|----------|------|
| 心跳槽位 | 节点运行态 / Redis 心跳槽位 | 用于脉冲图 |
| 心跳成功率 | 24h 心跳聚合 | 用于健康度统计 |
| 设备异常 | 设备台账 + 运行态 + 告警 | 只返回异常设备 |
| 系统状态 | 节点运行快照 / 分钟指标 | CPU、NPU、内存、存储、温度 |
| 当前任务 | 任务表最近运行中任务 | 展示节点当前承载任务 |

设备异常目标枚举：

| 枚举 | 含义 | 触发来源 |
|------|------|----------|
| `OFFLINE` | 设备离线 | 设备心跳超时或连接断开 |
| `STREAM_LOST` | 视频流中断 | 拉流失败或连续无帧 |
| `FRAME_DROP` | 高丢帧 | 丢帧率超过阈值 |
| `AUTH_FAILED` | 鉴权失败 | RTSP/设备凭据校验失败 |

#### 3.4.2 任务流水模型

任务流水继续使用现有任务主表和任务日志表：

| 数据 | 对应能力 | 说明 |
|------|----------|------|
| 任务统计 | `EdgeTaskStatisticsRespVO` | 今日下发、执行中、成功率、异常 |
| 任务分页 | `EdgeTaskRespVO` | 任务列表字段 |
| 任务详情 | `EdgeTaskDetailRespVO` | 下发策略、命令 Topic、Payload |
| 任务日志 | `EdgeTaskLogRespVO` | 事件、阶段、状态、进度、日志消息 |

### 3.5 关键流程

#### 3.5.1 心跳监控流程

1. 边缘节点按心跳周期上报心跳与系统指标。
2. 后端写入节点运行快照，并维护最近槽位状态。
3. 前端调用 `GET /edge/monitor/heartbeat/page`。
4. 页面展示节点名称、最后心跳、最近槽位、状态标签。
5. 若节点状态为 `WARNING` 或 `OFFLINE`，页面仍展示在心跳列表中。

#### 3.5.2 连接设备异常流程

1. 边缘节点或设备适配器上报设备状态、视频流状态、拉流错误或丢帧指标。
2. 后端归一化为设备异常记录，过滤正常设备。
3. 前端调用 `GET /edge/monitor/device-abnormal/page`。
4. 页面展示异常设备，并以 `statusLabel`、`metricText`、`abnormalReason` 帮助定位。
5. 异常恢复后，该设备从异常列表消失；历史可进入告警或日志域查询。

#### 3.5.3 系统状态预警流程

1. 边缘节点上报 CPU、NPU/GPU、内存、存储、温度等系统参数。
2. 后端按阈值生成 `systemStatus`、`healthLevel`、`recommendation`。
3. 前端调用 `GET /edge/monitor/system-status/page`，默认按 `npu` 排序。
4. 用户可切换 `sortBy=cpu|npu|temp`。
5. 页面沿用高负载预警表格样式，展示系统主要参数和处置建议。

#### 3.5.4 任务流水与日志流程

1. 用户进入任务流水页，前端并发加载任务统计和第一页任务流水。
2. 用户按关键词、任务类型、任务状态筛选，前端重置页码后重新查询。
3. 点击“日志”打开任务详情抽屉，前端加载任务详情和日志分页。
4. 对未完成任务可撤回，调用 `POST /edge/task/{taskKey}/cancel`。
5. 对失败或超时任务可重试，调用 `POST /edge/task/{taskKey}/retry`，后端返回新任务 Key。

| 旧设计 | 新目标 |
|--------|--------|
| 监控页展示注册链路跟踪 | 移除注册链路展示 |

## 6. 风险与回滚

- 风险：
  - 当前后端监控概览仍有 `registeringCount`，与新原型“心跳监控节点”语义不一致。
  - 当前源码未发现设备异常列表接口，直接前端落地会缺少后端契约。
  - 当前 `EdgeMonitorHighLoadRespVO` 未包含 `storageUsageRate`、`systemStatus`、`recommendation`，系统状态预警需要扩展。
  - 任务原型使用小写类型与状态，生产契约使用大写枚举，前端落地必须做映射。

  - 文档层面回滚：删除本次新增两份文档，并从 `docs/README.md` 移除索引行。
  - 实现层面回滚：保留旧 `EdgeMonitorController` 与 `EdgeTaskController`，生产页面继续使用当前接口。

  - 未读取实时 `/v3/api-docs`。
