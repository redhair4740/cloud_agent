- 后端开发 / AI 代码生成代理

- 后续测试、部署、运维人员

- 系统架构与链路设计

### 2.2 当前真实缺口

当前仓库虽然已有边缘节点台账，但仍缺少以下关键能力：

- 生产级 MQTT / EMQX 认证链路
- MQTT 上行事件统一入口与幂等处理
- 监控中心后端聚合接口
- 任务中心主数据模型与状态机
- 统一任务资源编排层
- 监控页与任务页前端实现
- 节点凭证安全模型升级

因此本方案不延续旧方案的“单文档堆叠式设计”，而采用分域、可落地、可演进的生产实现模型。

## 3. 核心设计目标

### 3.1 业务目标

系统必须实现以下目标：

1. 云端可管理边缘节点台账与设备凭证。
2. 边缘节点可通过 EMQX 与云端建立稳定连接。
3. 云端可接收并处理节点注册、心跳、快照、任务进度上报。
4. 云端可对节点进行统一任务下发。
5. 前端可查看监控总览、节点健康、任务列表、任务日志。

### 3.2 技术目标

系统必须满足以下技术目标：

1. 后端单体内按域分层，避免 controller 直连 mapper。
2. MQTT runtime 链路具备幂等、重试、审计、失败归档能力。
3. 节点凭证不长期保存明文。
4. 高频指标与静态台账分离，避免主表热更新。
5. 前端与后端接口契约严格对齐，前端不“猜字段”。

## 4. 总体架构

### 4.1 推荐生产架构

```text
Edge Node
  -> EMQX Broker
    -> HTTP Authentication
    -> Rule Engine / HTTP Sink
      -> Cloud Edge Runtime Ingest
        -> PostgreSQL / Redis
        -> Monitor Aggregation
        -> Task State Machine
  <- EMQX Publish API
Cloud Admin UI
  -> Cloud REST API
```

### 4.2 架构决策

本方案采用以下明确决策：

1. Cloud 不作为长期 MQTT 订阅消费者。
2. 上行事件通过 EMQX `HTTP Authentication + Rule Engine / HTTP Sink` 进入 Cloud。
3. 下行命令统一通过 EMQX `/api/v5/publish` 发布。
4. 监控与任务数据源以 Cloud 持久化结果为准，不以前端临时聚合为准。
5. 管理后台实时刷新采用 `REST + 轮询`，不以 WebSocket/SSE 作为 V1 主方案。

### 4.3 采用该架构的原因

- 更容易控制安全边界。
- 更容易做幂等处理与失败重放。
- 更适合当前单体后台代码结构。
- EMQX 与业务逻辑解耦，后续替换 broker 风险更小。
- 运维与审计更清晰。

## 5. 领域划分

后端所有新增能力统一归属 `cloud/vmesh-module-edge`，在模块内新增四个子域。

### 5.1 `node` 节点静态台账域

负责：

- 节点静态资料
- 设备凭证生命周期
- 节点预配置

### 5.2 `runtime` 节点运行时接入域

负责：

- EMQX 鉴权
- EMQX 事件接入
- 原始事件落库
- 幂等与重试

### 5.3 `monitor` 监控聚合域

负责：

- 节点状态快照
- 指标分钟聚合
- 健康分计算
- 告警生成与恢复

### 5.4 `task` 任务中心域

负责：

- 任务创建
- 命令封装
- 状态机推进
- 日志记录
- 取消与重试

### 5.5 `resource` 统一任务资源域

负责：

- OTA 资源
- 算法包资源
- 配置模板
- 命令模板

注意：`resource` 域是 `edge` 内部统一编排层，不直接绑定现有 AI/IoT 表结构。

## 6. 节点与凭证模型设计

### 6.1 设计原则

节点静态资料与凭证版本分离，避免以下问题：

- secret 被长期明文持久化
- 凭证轮换无法追踪
- 多版本凭证无法审计

### 6.4 凭证安全规则

1. secret 只在创建或轮换当次返回明文。
2. 数据库仅保存 `secret_hash`。
3. 查看凭证详情接口不返回明文 secret。
4. 凭证轮换必须生成新版本，不覆盖旧记录。
5. 旧凭证可配置立即废弃，或保留短暂过渡窗口。

## 7. MQTT / EMQX 设计

### 7.1 接入身份规则

```text
clientId = edge-node:{deviceId}
username = {deviceId}
password = secret
topicRoot = vmesh/edge/node/{deviceId}
```

### 7.2 Topic 规范

| 方向 | Topic | QoS | 用途 |
| --- | --- | --- | --- |
| 上行 | `{topicRoot}/register` | 1 | 注册与首次上线 |
| 上行 | `{topicRoot}/heartbeat` | 0 | 节点心跳与资源指标 |
| 上行 | `{topicRoot}/device/report` | 1 | 外设快照 |
| 上行 | `{topicRoot}/event` | 1 | 业务事件，V1 只支持 `task_progress` |
| 下行 | `{topicRoot}/command/{commandKey}` | 1 | 云端任务下发 |

### 7.3 MQTT 消息通用字段

所有业务消息 payload 默认包含：

- `message_id`
- `occurred_at`

说明：

- `message_id` 用于幂等去重。
- `occurred_at` 统一使用 13 位毫秒时间戳。

### 7.9 EMQX 接入方式

#### 鉴权

EMQX 调用 Cloud：

```text
POST /admin-api/edge/runtime/authenticate
```

Cloud 返回：

- `result`
- `is_superuser`
- `client_attrs`
- `acl`

#### 上行事件进入 Cloud

EMQX Rule Engine / HTTP Sink 调用：

```text
POST /admin-api/edge/runtime/ingest
```

请求必须带共享密钥头，例如：

```text
X-EMQX-SIGNATURE: <shared-secret-or-hmac>
```

#### 下行命令

Cloud 调用：

```text
POST {emqxApiBaseUrl}/publish
```

### 7.10 安全部署基线

V1 强制要求：

- 关闭匿名连接
- 生产环境启用 TLS Listener
- EMQX 管理 API 仅允许内网 Cloud 访问
- Webhook / HTTP Sink 必须做签名或共享密钥校验
- Cloud 必须校验来源 IP 白名单

## 8. runtime 运行时设计

### 8.1 设计原则

运行时链路必须具备：

- 幂等
- 可重试
- 失败归档
- 审计可追踪

### 8.2 `edge_runtime_event` 原始事件表

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `bigint` | 主键 |
| `node_id` | `bigint` | 节点 ID，可空 |
| `event_id` | `varchar(64)` | runtime 事件唯一号 |
| `source_type` | `varchar(32)` | 来源类型 |
| `event_type` | `varchar(64)` | `client.connected / client.disconnected / register / heartbeat / device.report / task.progress` |
| `message_id` | `varchar(64)` | MQTT 消息 ID |
| `client_id` | `varchar(128)` | 客户端 ID |
| `device_id` | `varchar(64)` | 设备身份 |
| `topic` | `varchar(255)` | MQTT Topic |
| `dedupe_key` | `varchar(128)` | 幂等键 |
| `occurred_at` | `timestamptz` | 业务发生时间 |
| `received_at` | `timestamptz` | Cloud 接收时间 |
| `payload_json` | `jsonb` | 原始 payload |
| `process_status` | `varchar(32)` | `PENDING / PROCESSING / DONE / RETRYABLE / FAILED / DISCARDED` |
| `retry_count` | `int4` | 重试次数 |
| `error_message` | `varchar(500)` | 错误信息 |
| `creator` | `varchar(64)` | 创建者 |
| `create_time` | `timestamptz` | 创建时间 |
| `updater` | `varchar(64)` | 更新者 |
| `update_time` | `timestamptz` | 更新时间 |
| `deleted` | `smallint` | 逻辑删除 |
| `tenant_id` | `bigint` | 租户 ID |

### 8.3 幂等键设计

- `register / heartbeat / device.report`：
  `tenantId + deviceId + messageId`
- `task.progress`：
  `tenantId + taskKey + messageId`
- `client.connected / client.disconnected`：
  `tenantId + eventId`

### 8.4 处理流程

1. controller 接收请求。
2. 校验签名、来源、基础结构。
3. 解析设备身份。
4. 生成幂等键。
5. 原始事件先落库。
6. 异步交给对应 handler 处理。
7. 成功则标记 `DONE`。
8. 可恢复失败标记 `RETRYABLE`。
9. 非法请求标记 `DISCARDED`。

### 8.5 handler 分工

- `ConnectionEventHandler`
- `RegisterEventHandler`
- `HeartbeatEventHandler`
- `DeviceReportEventHandler`
- `TaskProgressEventHandler`

每个 handler 只负责本域逻辑，禁止跨层乱调 mapper。

### 9.4 心跳健康度计算

默认心跳周期：

```text
20 秒
```

24 小时理论槽位数：

```text
86400 / 20 = 4320
```

计算公式：

```text
heartbeat_success_rate_24h = 成功槽位 / 4320 * 100
heartbeat_health_score = round(heartbeat_success_rate_24h)
```

### 9.5 综合健康分

加权规则：

- 心跳健康分：50%
- 延迟得分：30%
- 丢包率得分：20%

计算公式：

```text
health_score = round(heartbeatScore * 0.5 + latencyScore * 0.3 + packetLossScore * 0.2)
```

### 9.6 健康等级

| 分值 | 等级 |
| --- | --- |
| `>= 90` | `EXCELLENT` |
| `>= 75` | `GOOD` |
| `>= 60` | `FAIR` |
| `>= 40` | `POOR` |
| `< 40` | `CRITICAL` |

- `download`
- `verify`
- `install`

### 10.7 核心规则

1. `task_key` 是任务业务唯一号。
2. `message_id` 是单条事件唯一号。

4. 前端不得直接调用底层 MQTT 发送接口。
5. 所有任务必须经 `task dispatch` 统一创建。

## 13.3 鉴权实现

`EdgeRuntimeAuthService` 处理逻辑：

1. 根据 `username` 查询 `ACTIVE` 凭证。
2. 校验节点启用状态。
3. 校验 `clientId == edge-node:{deviceId}`。
4. 使用 `PasswordEncoder.matches` 比对 secret。
5. 生成 `client_attrs`。
6. 生成 per-device ACL。
7. 异步更新 `last_auth_at`。

### 13.4 ingest 接口实现

`EdgeRuntimeIngestController` 只负责：

- 头部校验
- DTO 校验
- 基础身份解析
- 调用 `EdgeRuntimeIngestService`

`EdgeRuntimeIngestService` 负责：

1. 生成 dedupe key。
2. Redis 预去重。
3. 原始事件落库。
4. 提交 processor。
5. 返回接收结果。

### 13.5 事件处理实现

#### `client.connected`

- 更新 `runtime_snapshot.connection_state = ONLINE`
- 写入 `last_connected_at`
- 如注册未完成，则注册状态推进到 `HANDSHAKING`

#### `client.disconnected`

- 节点置为 `OFFLINE`
- 写 `last_disconnected_at`
- 触发离线告警判定

#### `register`

- 更新 `registration_status = ACTIVATED`
- 写 `last_register_at`
- 初始化 heartbeat 参数

#### `heartbeat`

- 更新 snapshot 指标
- 写 Redis bitmap
- 写分钟聚合输入
- 推进运行状态

#### `device.report`

- 更新设备快照或后续 `edge_device` 同步信息

#### `task.progress`

- 查询任务
- 按状态机更新 `edge_task`
- 插入 `edge_task_log`

### 13.6 任务下发实现

`EdgeTaskDispatchService` 流程：

1. 校验节点存在且启用。
2. 校验资源存在且启用。
3. 校验资源类型与任务类型匹配。
4. 生成 `task_key`。
5. 组装 command envelope。
6. 创建 `edge_task`。
7. 调用 `EdgeTaskCommandPublisher` 发布到 EMQX。
8. 写任务日志。

### 13.7 发布器实现

`EdgeTaskCommandPublisher`：

- 使用 `RestTemplate` 或 `RestClient`
- 调 EMQX `/api/v5/publish`
- Header 使用 API Key
- 失败抛业务异常并记录 task log

### 13.8 调度任务

需要新增定时任务：

- `EdgeHeartbeatTimeoutJob`
  用于心跳超时离线判定
- `EdgeMetricAggregationJob`
  用于分钟级聚合
- `EdgeRuntimeRetryJob`
  用于失败事件重放
- `EdgeTaskTimeoutJob`
  用于超时任务收敛

## 17. 验收标准

### 17.1 节点接入验收

- 新节点创建成功
- 一次性凭证返回成功
- 节点可连接 EMQX
- register 后注册状态更新正确

### 17.2 监控验收

- heartbeat 能驱动 snapshot 更新
- 心跳健康度能计算
- 高负载排行可展示
- 离线告警与恢复可闭环

### 17.3 任务验收

- 云端可创建任务
- EMQX publish 成功
- 设备回传 task_progress
- 页面进度随之更新
- 失败任务可重试

### 17.4 安全验收

- 历史 secret 不可查询
- 节点禁用后无法认证
- 非法签名请求被拒绝
- 重复事件不会重复推进状态机

## 18. 风险与边界

### 18.1 已知风险

- 若设备端尚未统一 `task_progress` payload，则任务状态机需要同步适配。
- 若 EMQX 侧配置不可控，HTTP Sink / ACL 落地会受外部环境影响。
- 若现有 `edge_node` 表与当前页面已耦合较深，迁移到静态 + 运行态分离时需兼容一段时间。

### 18.2 明确不做

V1 不做以下事项：

- 批量多节点任务编排
- WebSocket / SSE 实时推送
- mTLS 证书认证
- 复杂流式日志订阅
- 多 broker 高可用切换逻辑

## 19. 默认值与假设

- V1 以单节点单任务为核心模型。
- V1 使用 `deviceId / secret + TLS + ACL` 作为认证基线。
- V1 允许 `resource` 域引用外部 AI / IoT 资源，但不直接依赖其表结构。
- V1 前端采用轮询实现实时感，不采用长连接推送。
- V1 监控历史以分钟聚合为主，不做秒级全量持久化。

## 20. 总结

这份方案的核心不是“把旧文档写得更长”，而是重新建立一套适合当前仓库落地的生产级边云协同模型：

- 静态台账与运行态解耦
- 凭证版本化与明文 secret 一次性返回
- runtime 入口统一幂等处理
- 监控中心以聚合与快照为主
- 任务中心以统一资源编排层与标准状态机为主
- 前后端契约可直接驱动代码生成

后续进入实现阶段时，应严格以本文档的表结构、接口定义、状态机和分层边界为准，不再以原型字段或旧方案细节作为实现依据。
