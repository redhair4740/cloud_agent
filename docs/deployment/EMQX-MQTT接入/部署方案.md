# EMQX MQTT 接入部署方案

## 1. 文档目标

- 适用环境：本地、dev、测试、生产 EMQX 接入。
- 部署对象：EMQX Broker、EMQX HTTP Authentication、EMQX Rule Engine / HTTP Sink、Cloud Edge Runtime。
- 当前状态：后端源码已存在鉴权、ingest、Topic 构造、任务下发发布逻辑；本文未执行真实 EMQX 联调。
- 验证口径：以 `EdgeRuntimeController`、`EdgeRuntimeServiceImpl`、`EdgeTopicService`、`EdgeRuntimeProperties`、`EdgeTaskServiceImpl` 为证据。

## 2. 架构与依赖

| 组件 | 责任 | 证据 |
|------|------|------|
| EMQX Broker | 边缘节点 MQTT 接入 | 部署环境提供 |
| EMQX HTTP Authentication | 调 Cloud 鉴权 | `POST /admin-api/edge/runtime/authenticate` |
| EMQX Rule Engine / HTTP Sink | 转发上行消息 | `POST /admin-api/edge/runtime/ingest` |
| Cloud Edge Runtime | 校验签名、Topic、凭证、硬件指纹并推进状态 | `EdgeRuntimeServiceImpl#ingest` |
| Cloud Task Service | 调 EMQX `/publish` 下发命令 | `EdgeTaskServiceImpl#publishCommand` |

| 配置项 | 示例值 | 说明 | 来源 |
|--------|--------|------|------|

## 4. Topic 规范

默认 `topicRoot` 由 `EdgeTopicService#buildDefaultTopicRoot` 生成：

```text
moon/{env}/tenants/{tenantId}/edge/{version}/nodes/{deviceId}
```

上行 Topic：

| 方向 | Topic | eventType | 用途 |
|------|-------|-----------|------|
| Edge -> Cloud | `{topicRoot}/lifecycle/register` | `register` | 注册与硬件指纹绑定 |
| Edge -> Cloud | `{topicRoot}/telemetry/heartbeat` | `heartbeat` | 心跳、资源指标 |
| Edge -> Cloud | `{topicRoot}/telemetry/device-report` | `device.report` | 外设/摄像头状态快照 |
| Edge -> Cloud | `{topicRoot}/events/task-progress` | `task_progress` 或 `task.progress` | 任务进度与结果回执 |

下行 Topic：

| 方向 | Topic | 用途 |
|------|-------|------|
| Cloud -> Edge | `{topicRoot}/commands/{commandKey}` | 云端任务下发 |

当前源码中的 ACL 由 `EdgeRuntimeServiceImpl#authenticate` 返回，允许边缘端 publish 上行 Topic、subscribe `{topicRoot}/commands/#`。

## 5. EMQX 配置步骤

1. 配置 HTTP Authentication：

```text
POST http://<cloud-host>:<cloud-port>/admin-api/edge/runtime/authenticate
```

请求体：

```json
{
  "clientid": "${clientid}",
  "username": "${username}",
  "password": "${password}",
  "peerhost": "${peerhost}",
  "proto_ver": "${proto_ver}"
}
```

2. 配置 Rule Engine / HTTP Sink：

```text
POST http://<cloud-host>:<cloud-port>/admin-api/edge/runtime/ingest
```

Header：

```text
Content-Type: application/json
X-EMQX-SIGNATURE: <VMESH_EDGE_INGEST_SHARED_SECRET>
```

Action 配置边界：

| 配置位置 | 推荐值 |
|----------|--------|
| HTTP Connector URL / Base URL | `http://<cloud-host>:<cloud-port>` |
| Rule Action Method | `POST` |
| Rule Action Path | `/admin-api/edge/runtime/ingest` |

不要把完整 URL 写到 Action Path 中，避免 EMQX 生成 `/http://...` 这类错误路径。

3. 推荐 HTTP Sink Body：

```json
{
  "eventType": "${payload.event_type}",
  "eventId": "${payload.message_id}",
  "messageId": "${payload.message_id}",
  "clientId": "${clientid}",
  "username": "${username}",
  "deviceId": "${username}",
  "hardwareFingerprint": "${payload.data.hardware_fingerprint}",
  "topic": "${topic}",
  "occurredAt": ${payload.occurred_at},
  "payload": ${payload}
}
```

模板要求：

- `occurredAt` 为数字时不要加引号。
- 不能渲染 `undefined`。
- `clientId` 必须透传 MQTT 客户端 ID。
- `deviceId` 应使用节点凭证身份，不要使用 Dashboard 管理账号。
- `register`、`heartbeat`、`device.report`、`task_progress` 必须携带硬件指纹。

## 6. 验证与回滚

验证命令/动作：

1. 创建或刷新边缘节点凭证，获取 `deviceId`、`secret`、`topicRoot`。
2. 使用 MQTT 客户端连接 EMQX，连接参数为 `clientId=edge-node:{deviceId}`、`username={deviceId}`、`password=secret`。
3. 发布 `{topicRoot}/lifecycle/register`。
4. 发布 `{topicRoot}/telemetry/heartbeat`。
5. 在管理端调用 `/edge/monitor/heartbeat/page` 和 `/edge/node/get` 查看状态。
6. 从 Cloud 下发任务，确认 EMQX 收到 `{topicRoot}/commands/{commandKey}`。

预期结果：

- 鉴权成功时 EMQX 获得 per-device ACL。
- ingest 成功时返回 `CommonResult<Boolean>`，节点状态推进。
- 心跳停止后，巡检任务在超时窗口后将节点降级为离线。

回滚方案：

- EMQX 侧禁用新增规则或恢复旧 Rule。
- Cloud 侧回退 `vmesh.edge.runtime.*` 环境变量。
- 任务下发失败时先停止新任务发布，保留节点台账和运行事件。

未验证项：

- 未执行生产 TLS、网络白名单、集群高可用验证。
