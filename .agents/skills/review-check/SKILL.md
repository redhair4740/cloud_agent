# Review 检查流程

## 触发场景

- 用户要求 review、评审、代码审查或回归检查。
- 需求已完成，需要上线前风险扫描。
- 存在跨模块、跨端或契约相关变更。

## 评审重点

- 行为回归、异常路径、权限、并发、数据一致性。
- 缺失测试或验证强度与风险不匹配。
- 请求/响应字段、枚举、错误码、分页参数前后端不一致。
- 模块边界破坏、跨域直接依赖或职责下沉到错误层级。

## 输出要求

- 先输出 Findings：严重级别、文件路径、问题说明、影响和建议修复。
- 未发现问题时明确写“未发现评审问题”，并补充残余风险或测试盲区。
- 未验证项必须显式标注“未验证”，不得写成“已修复/已通过”。

## 严重级别

- `S0`：阻断上线，存在数据损坏、安全或核心流程不可用风险。
- `S1`：高优先级缺陷，可能导致核心功能异常或明显回归。
- `S2`：中等问题，短期可上线但需排期修复。
- `S3`：低风险优化项，不影响主要功能正确性。
