# 前端开发流程

## 触发场景

- 页面开发、页面重构、交互优化、样式和可用性修复。
- 既有接口契约下的数据展示、表单提交、错误提示和状态反馈。
- 前端 API 调用封装、类型对齐、状态流转、组件复用。

## 执行流程

1. 确认当前需求能否在既有 OpenAPI 契约下完成。
2. 新增接口消费时补齐请求参数类型、响应类型和错误处理分支。
3. 页面状态分层管理：临时状态、表单状态、服务端数据状态不要混成单一大 store。
4. Mock 只放统一 Mock 层或统一 mock 目录，不散落到组件、Pinia action 或 API SDK 内部。
5. 原型只参考布局、信息层级、交互流程和功能点，不复用临时字段或代码。
6. 覆盖正常路径、空数据、权限不足和接口错误等典型失败路径。

## 注释要求

- 页面初始化与数据装载说明首次进入页面拉哪些接口、先后顺序、并发或串行原因。
- 查询条件与接口参数映射说明表单字段如何映射到 API params，哪些字段是前端派生、哪些直接透传。
- 轮询、自动刷新、分页、排序、弹窗、抽屉、上传、下载、剪贴板等流程说明触发链路、清理时机和失败兜底。
- 纯展示型类型不逐字段注释，仅对易误解字段、单位或枚举值补少量语义说明。

## 需要切换联动的情况

- 需要新增、删除、重命名接口字段或调整响应结构。
- 需要新增接口、合并接口或调整错误码、鉴权、租户隔离语义。
- 既有契约无法支撑页面需求，需要后端确认或改造。

## 交付要求

- 说明页面/组件、API 调用点、状态流转点和关键交互分支。
- 涉及接口调用时标注 Mock 是否保留、真实 API 是否切换。
- 标注正常路径、空数据、权限不足、接口错误等验证覆盖范围。
- 未执行真实页面或接口验证时，必须写“未验证”。
