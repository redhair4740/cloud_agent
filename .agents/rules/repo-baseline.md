# 仓库通用基线

## 沟通与证据

- 全程使用简体中文沟通、分析、注释与交付说明。
- 关键判断必须证据化：优先给出文件路径、方法名、配置键、命令、参数和真实行为。
- 结论需可复查，不用“应该”“可能”替代可执行动作。

## 代码注释规范

- 注释必须使用简体中文。
- 注释的目标是解释“意图、边界、取舍、失败路径与风险”，而不是复述代码字面含义。
- 禁止逐行注释、禁止对显而易见的赋值/判断写重复注释；优先写在“复杂代码块”之前，用 1-5 行把读者需要的上下文说清楚。
- 任何可能被误用的约束必须注释清楚：
  - 输入来源（请求参数/请求体/消息/定时任务/DB）
  - 安全边界（鉴权、签名、租户、权限）
  - 幂等/去重规则（去重键、冲突处理）
  - 状态机（状态含义、转移条件）
  - 时间与单位（秒/毫秒/百分比等）
  - 阈值与来源（为什么是这个阈值、来自配置还是常量）
- 注释必须保持“稳定”：不要写易过期的临时结论（例如“后续一定会改成 X”）；需要 TODO 时写清触发条件与验收标准。

## 验证口径

- 未执行真实验证时，必须明确标注“未验证”或“未执行测试”。
- 只有存在实际验证证据时，才可使用“已修复”“已通过”“已完成”。
- 禁止把静态分析、阅读代码、局部推断表述成已验证结论。
- 声称编译通过时，必须同时给出真实编译命令、执行范围与结果。

## 原型与实现边界

- 原型只用于参考布局结构、信息层级、交互流程和功能点范围。
- 不复用原型中的实现代码、伪代码、临时字段命名或技术细节。
- 原型与真实契约冲突时，以已确认契约为准，并明确记录差异。

## 协作与联动

- 只要需求涉及前后端契约、字段、错误语义、鉴权语义或联调验收，必须按联动任务处理。
- 禁止单边先改后通知另一侧兜底；跨端变更必须先对齐范围、验收口径和回滚思路。
- 跨端未达成一致前，不做不可逆实现决策。

## 风险边界

- 未经明确确认，不执行高风险操作：批量删除、批量重命名、回滚、外网真实服务调用、生产环境数据写操作。
- 禁止危险 Git 操作作为默认手段，例如硬重置、覆盖式回退、强推覆盖历史。
- 发现敏感信息时仅允许定位与脱敏说明，禁止在对话或文档中回显明文。

## 文档管理

- 项目文档按类型归入 `docs/` 下的子目录：`design/`、`governance/`、`deployment/`、`review-tracking/`。
- 每个类型目录下按 `yyyy-MM-dd` 创建日期子目录，不再在文件名中重复日期。
- 文档命名格式：`hh-mm-ss-类型标签-文档名称.md`，时间戳在前保证默认排序按时间排列。
- 新建文档必须使用创建时的实际本地时间生成 `hh-mm-ss`；除非用户明确指定时间或续写同时间戳交接文档，禁止默认使用 `00-00-00` 占位。
- 类型标签枚举：`设计方案`、`交接文档`、`治理评审`、`协作分析`、`部署方案`。
- 交接文档与对应设计方案放在同一类型目录的同一日期子目录下，使用相同时间戳。
- 新增文档后必须更新 `docs/README.md` 索引。
- `review-tracking/` 使用 `review清单-hh-mm-ss-文档名称-状态.md`。
- `review-tracking/` 状态后缀仅允许 `已完成`、`未完成`；状态变化时需同步重命名文件并更新索引链接。
