# 前端开发规则

## 技术栈

- Node.js 使用 20.x LTS。
- 构建工具使用 Vite 5.x。
- 前端框架使用 Vue 3.x。
- UI 组件库使用 Element Plus，状态管理使用 Pinia。
- 未经明确任务要求，不引入平行框架、替代状态库或第二套 UI 体系。

## API SDK

- 所有后端请求必须经由项目既有 API SDK/请求封装层，不在页面组件内散落直连请求。
- 新增接口调用必须同步补齐请求参数类型、响应类型和错误处理分支。
- 前端只做展示层转换，不在 SDK、页面或 store 中重写后端业务规则与权限规则。
- 发现契约缺失或不一致时，不猜字段硬编码兜底，切换到 API-First 联动流程。

## 状态与页面

- Store 只承载展示态、交互态与服务端数据缓存，不承载后端未声明的一致性规则。
- 页面临时状态、表单状态、服务端数据状态分层管理，避免单 store 过载。
- 状态命名、派生状态和异步 action 必须可读可追踪，避免隐式副作用链。

## 代码注释要求

- 注释必须使用简体中文，优先解释“数据流、状态流转、接口映射与失败兜底”，避免逐行复述模板/样式代码。
- 必须补注释的位置：
  - 页面初始化与数据装载：首次进入页面拉哪些接口、先后顺序、并发/串行原因。
  - 查询条件与接口参数映射：表单字段如何映射到 API params，哪些字段是前端派生、哪些直接透传。
  - 轮询/自动刷新：刷新间隔、触发条件、销毁清理、并发保护（避免重复请求）和错误退避策略（如果没有也要说明）。
  - 分页/排序：分页参数来源、回调触发链路、空数据与 loading 状态处理。
  - 弹窗/抽屉：打开、提交、关闭、回填、刷新父列表的时序与边界。
  - 浏览器能力与降级：剪贴板、下载、文件上传等能力失败时如何提示用户。
- 不强制对纯展示型 `VO` 类型逐字段写注释；仅对“易误解字段/单位/枚举值”补少量语义注释。

## Mock 边界

- Mock 仅用于开发阶段短期阻塞解除和界面占位验证，不作为长期依赖。
- Mock 只能落在统一 Mock 层或统一 mock 目录，禁止写入 Vue 组件、Pinia action 或 API SDK 内部。
- 接口可用后默认切真实 API，并清理仅服务该接口的失效 Mock；暂不清理需说明原因和期限。

## 交付要求

- 说明变更页面/组件、API 调用点、状态流转点和失败路径处理。
- 涉及接口调用时标注 Mock 是否保留、真实 API 是否切换。
- 明确已验证与未验证范围。
