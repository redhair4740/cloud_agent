# 后端开发流程

## 触发场景

- 后端接口实现、Service/Mapper/DO/VO 调整、后端配置和后端测试。
- 数据库增量 SQL、PostgreSQL 兼容、模块边界收敛。
- 不需要同步修改前端页面、状态、调用链即可验收的任务。

## 执行流程

1. 判断任务是否仅后端内部闭环；如涉及接口契约或前端消费，切换为 API-First 联动处理。
2. 确认需求归属域，优先落入现有业务域；不为局部方便创建平行模块。
3. 涉及 SQL 时确认增量脚本路径、迁移路径、历史数据影响、回滚思路和 PostgreSQL 行为。
4. 涉及鉴权、数据权限、租户隔离或审计时单列风险。
5. 按对象分层规则实现，避免 Controller 直接访问 Mapper 或承载复杂业务编排。
6. 选择最小可行验证：模块编译、局部单元测试或静态检查；未执行则明确标注。

## 注释要求

- 注释必须使用简体中文，优先写在复杂代码块之前，解释意图、边界、风险和失败路径。
- `Service` 入口方法用 3-10 行说明业务编排链路：读路径、写路径、依赖数据和关键副作用。
- 鉴权、签名、权限、租户隔离、幂等去重、状态机、事务边界、指标阈值等容易误用的位置必须补充边界说明。
- 不要求逐字段注释 VO/DO，仅对容易误解的字段补 1 行语义注释。

## 验证选择

- 后端代码默认偏向执行最小可行编译验证，不能只靠阅读代码判断。
- 低复杂度文案、常量、注解等小改动可不编译，但交付必须标注“未编译验证”。
- 中复杂度变更执行模块级编译，重点覆盖方法签名、类型转换、Mapper/DO/VO 字段、依赖注入、配置键、SQL 映射、TypeHandler 和序列化逻辑。
- 高复杂度变更除模块编译外，建议补充跨模块、数据库迁移、鉴权链路、启动装配、消息链路或第三方集成验证。
- 涉及 `<backend-dir>/<edge-module>` 时，优先执行：`cd "<backend-dir>" && mvn compile -pl "<edge-module>" -am`。
- 其他模块优先使用“目标模块 + `-am`”的最小编译范围。

## 需要切换联动的情况

- 新增接口或调整接口路径、方法、请求参数、响应字段、枚举、错误码、鉴权语义。
- 后端变化会影响前端页面渲染、状态管理、交互流程或联调验收。
- 验收标准要求前后端同测或端到端回归。

## 交付要求

- 说明目标、落地文件、影响模块、边界决策和回归风险。
- 涉及数据库变更时，说明 SQL 落地路径、迁移路径、回滚思路和历史数据影响。
- 给出真实验证命令、执行范围与结果；未执行时明确“未验证/未执行测试/未编译验证”。
- 不把业务逻辑塞入启动装配层、通用基础设施层或错误域模块。
