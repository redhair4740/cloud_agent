# Portainer 部署发布

## 触发场景

- 设计或修订后端、前端 Docker/Portainer 发布方案。
- 编写 Jenkins 构建、Harbor 镜像推送、Portainer Stack 部署步骤。
- 排查部署资产、环境变量、镜像 tag、nginx 配置或回滚流程。

## 核心原则

- 不默认修改业务代码；部署资产与业务实现分开。
- 前后端部署资产各自管理，优先落在对应工程的 `deploy/` 目录。
- 密钥、数据库、Redis、MQTT、对象存储等敏感配置只通过环境或平台配置注入。
- 未真实构建、推送或启动时，不得写“已验证”。

## 工作流

1. 核对入口：后端 `pom.xml`、server 模块、Dockerfile、配置文件；前端 `package.json`、环境文件、请求配置。
2. 判断旧部署资产是否退役：旧 compose、脚本、镜像名、网络和卷是否仍被引用。
3. 拆分前后端发布边界：后端镜像、前端镜像、共享网络、环境变量、外部依赖。
4. 明确敏感配置来源：密钥、数据库、Redis、MQTT、对象存储等只通过环境或平台配置注入。
5. 输出 Portainer Stack、Jenkins/Harbor 步骤、首发操作清单和回滚检查清单。
6. 写清验证边界：本地构建、镜像构建、容器启动、健康检查、页面访问、接口联通。

## 建议产物

- 后端部署资产：`<backend-dir>/deploy/`
- 前端部署资产：`<frontend-dir>/deploy/`
- 发布检查清单：首发前置条件、发布步骤、验证步骤、回滚触发条件和回滚动作。
- 如需生成项目文档，按项目既有文档命名规则落位。

## 禁止事项

- 不把生产密钥写入仓库。
- 不调用生产 API 或写生产数据，除非用户明确确认。
- 不为部署方便改业务代码或绕过认证、租户、审计边界。
- 未真实构建、推送或启动时，不得写“已验证”。
