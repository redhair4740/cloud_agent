# 文档模板使用流程

## 触发场景

- 用户要求生成设计方案、协作分析、治理评审、部署方案或交接文档。
- 用户要求记录 Bug 修复、重构计划、评审清单或 review 跟踪项。
- 已完成实现，需要补充可归档的说明、验收标准、风险和未验证项。

## 模板选择

| 场景 | 模板 |
|------|------|
| 通用设计方案或联动方案 | `templates/design-doc.md` |
| API 契约文档 | `templates/api-doc.md` |
| 部署方案或运维方案 | `templates/deployment-doc.md` |
| Bug 修复工单 | `templates/bug-fix.md` |
| 重构工单 | `templates/refactor.md` |
| 交接文档 | `templates/handover-doc.md` |
| Review 跟踪清单 | `templates/review-checklist.md` |

需要示例且 `examples/` 下存在同名 `.example.md` 文件时再读取；不需要示例或示例不存在时不要额外加载。

## 使用流程

1. 先确认文档类型、目标读者、关联任务和是否涉及接口契约。
2. 按文档治理规则判定 `lifecycle`：`设计方案`、`API文档`、`部署方案` 优先作为活文档；`交接文档`、`治理评审`、`协作分析`、`review清单`、`Bug修复工单`、`重构工单`、`索引`、`测试数据说明` 作为记录文档。
3. 按判定结果执行“活文档流程”或“记录文档流程”。
4. 读取对应模板，只填充与任务相关的章节，不制造无法验证的内容。
5. 涉及接口契约时写清 OpenAPI 来源、接口状态、确认人和未确认项。
6. 交付前检查“已验证/未验证/风险/回滚/下一步”是否完整。

## 活文档流程

1. 先查 `docs/README.md` 主题索引，再查对应类型目录下的主题目录。
2. 找到同主题且 `status: active` 的活文档时，更新原文件；同步 `last_updated`，并在“变更日志”追加 `日期 / 变更摘要 / 变更来源`。
3. 未找到同主题活文档，或主题边界不同，创建 `docs/<类型目录>/<主题短名>/<类型标签>.md`。
4. 发现同主题旧文档需要迁移时，在新活文档 `supersedes` 填旧路径；在旧文档 frontmatter 补 `status: archived` 与 `superseded_by`。
5. 新增、迁移或更新活文档后，同步维护 `docs/README.md` 主题索引。

## 记录文档流程

1. 读取当前本地时间，按 `yyyy-MM-dd/hh-mm-ss` 落位；不得默认使用 `00-00-00` 占位。
2. `review清单` 需根据是否存在未完成项选择文件名状态后缀：`已完成` 或 `未完成`。
3. 记录文档关联某个主题时，将对应活文档路径写入 frontmatter `related`，并挂到 `docs/README.md` 对应主题索引。
4. 记录文档不重写历史结论；若后续状态变化，只补充完成说明、验证结果、归档元数据或索引修正。

## Frontmatter 填写

- 所有文档至少填写：`lifecycle`、`topic`、`type`、`created`、`status`。
- 活文档额外填写：`last_updated`、`supersedes`、`related`。
- 字段为空时保留空列表或空值，不要填入示例业务名。

## 输出要求

- 文档正文必须保留真实证据：文件路径、接口路径、命令、配置键或行为说明。
- 未执行验证时必须写“未验证/未执行测试/未编译验证”。
- Review 清单状态后缀只能使用 `已完成` 或 `未完成`。
- 记录文档不重写历史结论；允许补充状态、验证结果、归档元数据、完成说明和索引修正。
- 不把模板示例里的业务名、路径或状态当成当前任务事实。
