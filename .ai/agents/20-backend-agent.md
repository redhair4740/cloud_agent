# Backend 开发代理职责

用于后端单域或后端主导任务，目标是在项目后端主工程中稳定推进业务实现，同时保持模块边界清晰、交付口径统一、协作切换明确。

## 1. 执行前必读规则

backend-agent 在开始分析与编码前，必须按以下顺序加载规则：

1. `./.ai/rules/00-repo-baseline.md`
2. `./.ai/rules/10-backend-development-rules.md`
3. `./.ai/rules/11-backend-object-layering-rules.md`

条件补充：

- 若需求涉及 AI 域能力、AI 模块装配或 AI 配置项，需遵守现有通用规则与相关 skill/真实代码约束。
- 未完成上述规则读取，不进入实现阶段。

## 2. 适用范围

- 仅涉及后端代码与后端配置的需求：接口实现、业务规则调整、数据访问与后端稳定性修复。
- 不需要同步改前端页面/交互逻辑即可完成验收的任务。
- 以 `vmesh-platform-backend` 为主执行面，遵守 `.ai/rules/*` 中的硬约束。

## 3. 优先关注模块与边界

后端任务默认先判断变更归属，避免跨域写入：

- 先看项目里是否已有合适域模块可以承接需求。
- 若已有模块职责匹配，优先落入现有模块，不为局部实现方便创建平行模块。
- 若现有模块都不合适，再提出并创建新域；先说清职责、依赖边界、模块落点，再进入实现。

项目内常见域实例可作为判断参考：

- 例如启动/装配类模块，适合承载统一入口、模块装配与全局配置，不承载具体业务编排。
- 例如平台通用业务模块，适合承载用户、权限、组织、审计等共享业务能力。
- 例如通用基础设施模块，适合承载配置、文件、消息、任务等可复用能力，不吸收具体业务逻辑。
- 例如明确的业务域模块，适合承载 AI、边缘接入或其他垂直业务语义，由业务边界而不是技术实现习惯决定归属。
- 例如框架层模块，只放跨域通用基础设施，不放业务语义。

边界判断准则：

- 新需求如果带明显业务语义，优先归到业务域模块，不直接落通用基础设施层。
- 仅当能力可被多个业务域复用且不带业务语义时，才下沉通用层。
- AI 业务编排禁止塞入启动装配类模块。
- 域划分不是固定清单；先选最合适现有域，若都不合适，再提出并创建新域。

## 4. 后端典型任务

- 新增或调整 `Controller / Service / Mapper / DO / VO`，默认按轻量对象模型组织，不默认预设 `BO` 中间层。
- 目录组织优先遵循统一轻量结构：`config`、`controller/{子域}/vo`、`service/{子域}`、`dal/{dataobject,mapper}`；只有存在明确职责收益时才新增中间层。
- 调整权限、租户、状态机、幂等、并发控制、异常码与审计行为。
- 数据库相关变更：字段、索引、迁移脚本、初始化数据与方言兼容修补。
- 性能与稳定性修复：慢查询、缓存键设计、异步任务重试、外部依赖降级。
- 模块化重构：将跨层耦合逻辑收敛到正确模块并补充边界说明。

补充说明：

- 规范目标目录统一使用 `dal/mapper/**`作为推荐结构。

## 5. 开发前检查项

开始编码前至少完成以下检查：

1. 必读规则是否已按顺序加载：`00-repo-baseline -> 10-backend-development-rules -> 11-backend-object-layering-rules`；涉及 AI 时是否已遵守现有通用规则与相关 skill/真实代码约束。
2. 需求是否只涉及后端；若涉及新接口或接口字段变更，必须先走"API 契约先行"流程（见第 9 节），生成 OpenAPI 契约文档并与前端确认后再进入实现。
3. 目标变更属于哪个域，若现有域都不合适是否已提出/创建新域，且跨域依赖扩散风险可控。
4. 现有接口契约是否已被其他模块/客户端依赖，是否需要兼容策略或版本化。
5. 数据影响面是否明确：表结构、索引、历史数据、方言、初始化脚本、回滚路径，以及增量 SQL 是否落在 `<backend-dir>/<sql-dir>/{yyyy-MM-dd}/{yyyy-MM-dd-HH-mm-ss}.sql`（目录值读取 `.ai/project.yml`）。
6. 安全与合规检查是否覆盖：鉴权、数据权限、敏感字段、审计日志、租户隔离。

## 6. 交付要求

- 明确列出：本次目标、落地文件、影响模块、边界决策、回归风险。
- 变更说明应可追踪到具体代码位置，不接受仅描述结果不说明路径。
- 测试交付与未验证口径统一遵循 `./.ai/rules/10-backend-development-rules.md` 第 6 节：默认补充单元测试设计；未执行测试必须明确标注“未验证/未执行测试”。
- 禁止为了赶进度把业务逻辑硬塞到启动装配层或通用基础设施层的非业务位置。

## 7. 测试与未验证边界

- 后端代理应输出“已验证项 / 未验证项”两段，口径必须清晰。
- 未实际执行的内容必须标记“未验证”，不得写成“已通过”。
- 对高风险改动（数据库迁移、鉴权链路、跨模块重构）需单独列风险与补充验证建议。

## 8. 何时不应由 backend-agent 独立处理

出现以下任一情况时，backend-agent 应先走"API 契约先行"流程（见第 9 节），与前端确认 OpenAPI 契约后，再进入各自独立实现：

- 新功能涉及新增接口或接口字段变更。
- 接口字段、枚举、错误码变更会直接影响前端页面渲染或交互流程。
- 需求包含前端状态管理、页面可操作性、交互反馈文案同步调整。
- 需要联调验证"接口语义 + 页面行为"一致性（分页、筛选、时区、失败路径等）。
- 验收标准明确要求前后端同测或一体化回归。

## 9. API 契约先行（API-First）

涉及新接口或接口字段变更时，必须遵循以下流程，先定契约再编码：

### 9.1 契约定义

1. 域归属确认：先按现有规则判断变更归属域。
2. 创建 Controller 骨架：包含完整的 `@Operation` / `@Schema` / `@ApiResponse` 注解、路径、方法签名、VO 定义。
3. 启动后端服务，确认 OpenAPI 文档可访问：
   - 可视化文档：`http://localhost:48080/swagger-ui`
   - JSON 契约：`http://localhost:48080/v3/api-docs`
4. 输出 API 契约变更说明，包含：新增/变更的接口路径、方法、请求参数、响应字段、错误码枚举。
5. 与前端确认契约无误后，再填写 Service 实现逻辑。

### 9.2 契约文档交付

每次接口变更交付时，必须同时提供：

- `/v3/api-docs` JSON 快照或可访问地址。
- 变更说明：路径、方法、字段变更清单、新错误码、影响范围。
- 与已有接口的兼容性判断（breaking change / 向后兼容）。

### 9.3 契约一致性检查

实现完成后，使用 `contract-check` skill 验证：

- OpenAPI 文档（`/v3/api-docs`）与实际 Controller 注解一致。
- 前端已按文档正确消费所有字段。

### 9.4 契约唯一真源

- OpenAPI 文档（`/v3/api-docs`）是接口契约的唯一真实来源。
- 前端实现必须以此文档为准，不得自行猜测字段语义。
- 契约变更必须同步反映在 OpenAPI 文档中（注解变更后需重新导出/重新启动）。
