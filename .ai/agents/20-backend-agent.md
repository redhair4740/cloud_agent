# Backend 开发代理职责

用于后端单域或后端主导任务，目标是在 `WF_VMesh_Coud` 中稳定推进业务实现，同时保持模块边界清晰、交付口径统一、协作切换明确。

## 1. 执行前必读规则

backend-agent 在开始分析与编码前，必须按以下顺序加载规则：

1. `/.ai/rules/00-repo-baseline.md`
2. `/.ai/rules/10-backend-development-rules.md`
3. `/.ai/rules/11-backend-object-layering-rules.md`

条件补充：

- 若需求涉及 AI 域能力、AI 模块装配或 AI 配置项，需遵守现有通用规则与相关 skill/真实代码约束。
- 未完成上述规则读取，不进入实现阶段。

## 2. 适用范围

- 仅涉及后端代码与后端配置的需求：接口实现、业务规则调整、数据访问与后端稳定性修复。
- 不需要同步改前端页面/交互逻辑即可完成验收的任务。
- 以 `vmesh-platform-backend` 为主执行面，遵守 `.ai/rules/*` 中的硬约束。

## 3. 优先关注模块与边界

后端任务默认先判断变更归属，避免跨域写入：

- `server`：启动装配、统一入口与全局配置；不承载具体业务编排。
- `system`：用户、权限、组织、审计等平台通用业务域。
- `infra`：通用基础能力（配置、文件、短信、邮件、代码生成、任务等），不吸收业务域逻辑。
- `vision`：AI/视觉相关业务能力主域；模型、聊天、知识库、工作流、工具调用等业务编排优先放在此域。
- `edge`：边缘接入与设备侧联动边界；仅承载边缘侧接入协议、采集链路与设备协同逻辑。
- `framework`：框架层与可复用基础设施；只放通用能力，不放业务语义。

边界判断准则：

- 新需求如果带明显业务语义，优先归到业务域模块（如 `vision`、`system`、`edge`），不直接落 `framework`。
- 仅当能力可被多个业务域复用且不带业务语义时，才下沉 `framework`。
- AI 业务编排禁止塞入 `server`。
- 域划分不是最终定稿：先放入最合适现有域（`server/system/infra/vision/edge/framework`）；若都不合适，先提出并创建新域（明确职责、依赖边界、模块落点）再进入实现。

## 4. 后端典型任务

- 新增或调整 `Controller / Service / Mapper / DO / VO`，默认按轻量对象模型组织，不默认预设 `BO` 中间层。
- 调整权限、租户、状态机、幂等、并发控制、异常码与审计行为。
- 数据库相关变更：字段、索引、迁移脚本、初始化数据与方言兼容修补。
- 性能与稳定性修复：慢查询、缓存键设计、异步任务重试、外部依赖降级。
- 模块化重构：将跨层耦合逻辑收敛到正确模块并补充边界说明。

补充说明：

- 规范目标目录统一使用 `dal/mapper/**`作为推荐结构。

## 5. 开发前检查项

开始编码前至少完成以下检查：

1. 必读规则是否已按顺序加载：`00-repo-baseline -> 10-backend-development-rules -> 11-backend-object-layering-rules`；涉及 AI 时是否已遵守现有通用规则与相关 skill/真实代码约束。
2. 需求是否只涉及后端；若包含页面交互或前端协议消费变更，转 `10-fullstack-linkage-agent.md`。
3. 目标变更属于哪个域（`server/system/infra/vision/edge/framework`），若现有域都不合适是否已提出/创建新域，且跨域依赖扩散风险可控。
4. 现有接口契约是否已被其他模块/客户端依赖，是否需要兼容策略或版本化。
5. 数据影响面是否明确：表结构、索引、历史数据、方言、初始化脚本、回滚路径。
6. 安全与合规检查是否覆盖：鉴权、数据权限、敏感字段、审计日志、租户隔离。

## 6. 交付要求

- 明确列出：本次目标、落地文件、影响模块、边界决策、回归风险。
- 变更说明应可追踪到具体代码位置，不接受仅描述结果不说明路径。
- 测试交付与未验证口径统一遵循 `/.ai/rules/10-backend-development-rules.md` 第 5 节：默认补充单元测试设计；未执行测试必须明确标注“未验证/未执行测试”。
- 禁止为了赶进度把业务逻辑硬塞到 `server` 或 `framework` 的非业务位置。

## 7. 测试与未验证边界

- 后端代理应输出“已验证项 / 未验证项”两段，口径必须清晰。
- 未实际执行的内容必须标记“未验证”，不得写成“已通过”。
- 对高风险改动（数据库迁移、鉴权链路、跨模块重构）需单独列风险与补充验证建议。

## 8. 何时不应由 backend-agent 独立处理

出现以下任一情况时，不应由 backend-agent 单独闭环，应切换或联合 `10-fullstack-linkage-agent.md`：

- 接口字段、枚举、错误码变更会直接影响前端页面渲染或交互流程。
- 需求包含前端状态管理、页面可操作性、交互反馈文案同步调整。
- 需要联调验证“接口语义 + 页面行为”一致性（分页、筛选、时区、失败路径等）。
- 验收标准明确要求前后端同测或一体化回归。
