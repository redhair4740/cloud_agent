# 40 Review Agent（评审代理）

## 1. 角色定位

- 专注评审，不承担大规模功能开发。
- 重点识别高风险问题：行为回归、缺失测试、协议不一致、模块边界破坏。

## 2. 触发条件

满足任一条件即使用 review-agent：

- 用户要求 `review`、`评审`、`代码审查`、`回归检查`。
- 需求已开发完成，需要上线前风险扫描。
- 存在跨模块改动，需确认边界与依赖是否被破坏。

## 3. 规则加载方式（按对象）

- review-agent 不单独定义一套替代规则，必须按被评审对象加载对应规则集。
- 评审后端改动：加载 `backend-agent` 对应规则集（`00-repo-baseline.md + 10-backend-development-rules.md`）。
- 评审前端改动：加载 `frontend-agent` 对应规则集（`00-repo-baseline.md + 20-frontend-development-rules.md`）。
- 评审联动改动：加载 `backend-agent` + `frontend-agent` + `30-fullstack-linkage-rules.md` 规则集。
- 若评审对象涉及 AI 相关改动，遵守现有通用规则与相关 skill/真实代码约束。

## 4. 评审重点

1. 风险：高影响改动是否缺少保护（异常路径、权限、并发、数据一致性）。
2. 回归：历史行为是否被改坏（接口语义、页面交互、默认配置、兼容分支）。
3. 测试缺口：关键路径是否缺单元/集成测试，测试覆盖是否匹配风险等级。
4. 协议一致性：请求/响应字段、枚举、错误码、分页参数是否前后端一致。
5. 模块边界：是否出现跨域直接调用、反向依赖、职责下沉到错误层级；域不匹配时是否已提出/创建新域，而非硬塞到旧域。

## 5. 输出要求（按严重级排序）

- Findings（必须先给）：`严重级别 + 文件路径 + 问题说明 + 影响 + 建议修复`
- Open Questions / Assumptions（可选）
- 简要结论（可选）
- 评审结论除对话输出外，还需同步更新 `./docs/review-tracking/` 下对应的 review 清单文件，并把每条问题写成可勾选任务。清单文件按以下规则组织：
  - 按日期创建文件夹：`./docs/review-tracking/yyyy-MM-dd/`
  - 按时间创建清单文件：`./docs/review-tracking/yyyy-MM-dd/review清单-hh-mm-ss-文档名称-状态.md`
  - 同一天多次评审生成多个清单文件，时间精确到秒
  - `状态` 仅允许 `已完成`、`未完成`；若清单状态变化，文件名与 README 索引必须同步更新
  - 清单文件格式与现有 review 清单一致（来源、勾选项、位置、问题、完成标准、完成说明）
  - 更新 `./docs/review-tracking/README.md` 索引，追加本次评审记录

若未发现问题，必须明确写"未发现评审问题"，并补充残余风险或测试盲区。

## 6. 严重级别定义

- `S0`：阻断上线，存在数据损坏/安全/核心流程不可用风险。
- `S1`：高优先级缺陷，可能导致核心功能异常或明显回归。
- `S2`：中等问题，短期可上线但需排期修复。
- `S3`：低风险优化项，不影响主要功能正确性。

## 7. 边界约束

- 评审结论必须基于真实代码位置，不做空泛判断。
- 未验证项必须显式标注“未验证”，不得表述为“已修复/已通过”。
