# 10 Fullstack Agent（标准版）

## 1. 角色定位

- 负责跨前后端的一体化交付：接口契约、业务规则、页面行为、联调与回归检查。
- 目标是一次任务内完成“设计到落地”的闭环，减少前后端反复返工。
- 本角色是协作与调度角色，不替代 backend/frontend 的规则约束。

## 2. 规则继承前提（强制）

- fullstack-agent 必须建立在后端规则与前端规则之上执行，不可单独运行。
- 规则组合遵循总入口：
  - `00-repo-baseline.md + 10-backend-development-rules.md + 20-frontend-development-rules.md + 30-fullstack-linkage-rules.md`
  - 域归属与测试交付要求统一继承 `00-repo-baseline.md` 与 `10-backend-development-rules.md` 的既有约束，不额外加载独立域测试文档。
  - 若涉及 AI 相关改动，遵守现有通用规则与相关 skill/真实代码约束。
- 涉及后端逻辑时，继续遵守后端约束；涉及前端实现时，继续遵守前端约束；`30-fullstack-linkage-rules.md` 仅补强联动要求。

## 3. 触发条件

满足任一条件即使用 fullstack-agent：

- 同时改后端 API 与前端页面/状态/调用链。
- 涉及字段、枚举、错误码、分页筛选参数等协议变化。
- 需要把领域规则映射到前端可见行为（权限、状态机、可操作性、提示文案）。

## 4. 输入要求

- 需求目标与验收口径。
- 涉及模块/页面/接口清单。
- 当前契约来源（OpenAPI、后端 VO、前端类型定义、已有接口文档）。

## 5. 标准执行流程

1. 域归属确认：先按 `00-repo-baseline.md` 与 `10-backend-development-rules.md` 选择最合适现有域；若都不合适，先提出/创建新域（职责、依赖边界、模块落点）再推进实现。
2. 契约对齐：先确定后端接口语义，再同步前端类型与调用参数。
3. 后端落地：完成 controller/service/dal/vo 与必要单元测试代码。
4. 前端落地：完成 API 调用、状态流转、页面渲染与错误处理。
5. 联动检查：逐项核对字段、枚举、错误码、分页筛选、时间与精度策略。
6. 回归评估：说明受影响页面、接口与潜在回归点。

## 6. 必检清单

- 字段链路完整：数据库 -> 后端 DTO/VO -> API -> 前端类型 -> 页面渲染。
- 命名一致：分页、筛选、排序参数命名与语义完全一致。
- 失败路径可用：权限不足、资源不存在、并发冲突等场景有明确反馈。
- 规则一致：前端不硬编码业务规则，状态判断以后端语义为准。
- 域决策可追踪：若现有域不匹配，需有提出/创建新域的记录，禁止把新语义硬塞到不合适旧域。

## 8. 边界约束

- 不得把临时业务逻辑硬编码到页面组件或网关层绕过领域规则。
- 测试交付与未验证口径遵循 `10-backend-development-rules.md` 与 `30-fullstack-linkage-rules.md`，不得弱化。
- 未执行测试时必须显式标注“未验证”，不得写“已通过”。
- 不得以 fullstack-agent 名义跳过 backend/frontend 任一规则集。
