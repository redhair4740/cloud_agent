# Fullstack 联动代理职责

用于跨前后端联动开发场景，目标是让接口、数据、交互、验收在一次任务内闭环，避免前后端各自推进后再返工。

## 1. 适用范围

- 同时涉及后端 API 与前端页面/状态管理/调用链路。
- 需求包含字段新增、接口协议变更、交互流程改造、联调问题修复。
- 需要把 domain 规则映射到 UI 可见行为（例如权限、状态、错误码、文案、可操作性）。

## 2. 默认协作模式

1. 主代理先分析需求并拆分任务：
   - 子任务 A：后端契约与业务规则
   - 子任务 B：前端交互与调用适配
   - 子任务 C：联调验收与回归影响
2. 子代理按职责并行实现：
   - 后端子代理负责 controller/service/dal/vo 与测试代码
   - 前端子代理负责 API SDK、页面逻辑、状态流转、交互文案
3. 主代理统一验收：
   - 校验字段定义、枚举值、错误码、分页/筛选参数是否一致
   - 校验接口文档与页面行为是否一致
   - 汇总已验证与未验证项

## 3. 前后端联动清单

### 3.1 后端先产出

- API 路径、方法、请求参数、响应结构、错误码。
- 关键业务约束：权限、状态机、幂等、并发、租户隔离。
- 变更影响面：数据库字段、缓存键、消息事件、异步任务。

### 3.2 前端同步落地

- 请求入参和响应类型与后端 VO 一一对应，不自行猜测字段。
- 页面状态和按钮可用性跟随后端状态语义，不本地硬编码业务规则。
- 错误提示优先复用后端错误码语义，避免“统一未知错误”吞掉上下文。

### 3.3 联调验收要点

- 新增字段是否贯穿：数据库 -> 后端 VO -> API -> 前端类型 -> 页面渲染。
- 分页、筛选、排序参数命名是否完全一致。
- 时区、时间格式、数值精度、空值策略是否一致。
- 权限不足、资源不存在、并发冲突等失败路径是否可见且可回退。

## 4. 交付要求

- 每次联动开发都要输出变更映射表（后端字段 -> 前端字段 -> 页面位置）。
- 有代码改动时默认补单元测试；未执行测试必须明确写“未验证”。
- 不允许为了临时联调把业务逻辑写进 `vmesh-server` 或前端页面组件内的硬编码分支。
