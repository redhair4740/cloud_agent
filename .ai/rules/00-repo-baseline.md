# Rule: 仓库通用基线

本文件仅定义跨前后端通用规则；后端专属规范放 `10-backend-development-rules.md`，前端专属规范放 `20-frontend-development-rules.md`。

## 1. 沟通与证据表达

- 全程使用简体中文沟通（需求澄清、分析、变更说明、交付说明）。
- 关键判断必须证据化：优先给出 `文件路径 + 方法/配置键 + 命令/参数 + 真实行为`，避免无依据结论。
- 结论需可复查，不使用“应该/可能”替代可执行动作。

## 2. 未验证口径

- 未执行真实验证时，必须明确标注“未验证”或“未执行测试”。
- 只有存在实际验证证据时，才可使用“已修复”“已通过”“已完成”。
- 禁止把静态分析、阅读代码、局部推断表述成已验证结论。

## 3. 原型与实现边界

- 原型只用于参考布局结构、信息层级、交互流程、功能点范围。
- 不复用原型中的实现代码、伪代码、临时字段命名或技术细节。
- 原型与真实契约冲突时，以已确认契约为准，并明确记录差异。

## 4. 协作与联动边界

- 只要需求涉及前后端契约、字段、错误语义、鉴权语义或联调验收，必须按联动任务处理。
- 禁止单边先改后通知另一侧兜底；跨端变更必须先对齐范围、验收口径和回滚思路。
- 跨端未达成一致前，不做不可逆实现决策。

## 5. Skill 使用约束

- 新增能力、脚手架、规则体系或大改动前，先提出 skill 方案（目标、边界、产物），确认后再生成。
- 优先复用已有 skill 与仓库既有规则，避免重复造轮子与平行规范。

## 6. 通用风险边界

- 未经明确确认，不执行高风险操作：批量删除/重命名、回滚、外网真实服务调用、生产环境数据写操作。
- 禁止危险 Git 操作（如 `git reset --hard`、`git checkout -- <file>`、强推覆盖历史）作为默认手段。
- 发现敏感信息时仅允许定位与脱敏说明，禁止在对话或文档中回显明文。
