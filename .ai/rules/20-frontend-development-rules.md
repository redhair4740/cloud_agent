# Rule: 前端开发强制规则

本规则只定义前端技术、API SDK、状态管理、Mock 和验证边界。接口契约或联动流程见 `30-fullstack-linkage-rules.md`。

## 1. 技术栈

- Node.js 使用 20.x LTS。
- 构建工具使用 Vite 5.x。
- 前端框架使用 Vue 3.x（Composition API）。
- UI 组件库使用 Element Plus，状态管理使用 Pinia。
- 未经明确任务要求，不引入平行框架、替代状态库或第二套 UI 体系。

## 2. API SDK

- 所有后端请求必须经由项目既有 API SDK/请求封装层，不在页面组件内散落直连请求。
- 新增接口调用必须同步补齐请求参数类型、响应类型和错误处理分支。
- 前端只做展示层转换，不在 SDK、页面或 store 中重写后端业务规则与权限规则。
- 发现契约缺失或不一致时，不猜字段硬编码兜底，按 `30-fullstack-linkage-rules.md` 切换 API-First。

## 3. Pinia 与页面状态

- Store 只承载展示态、交互态与服务端数据缓存，不承载后端未声明的一致性规则。
- 页面临时状态、表单状态、服务端数据状态分层管理，避免单 store 过载。
- 状态命名、派生状态和异步 action 必须可读可追踪，避免隐式副作用链。

## 4. 原型参考

- 原型只参考布局结构、信息层级、交互流程和功能点范围。
- 不复用原型代码、伪代码、临时字段命名或技术实现细节。
- 原型与真实接口契约冲突时，以已确认 OpenAPI 契约为准，并记录差异。

## 5. Mock 边界

- Mock 仅用于开发阶段短期阻塞解除和界面占位验证，不作为长期依赖。
- 接口联调状态读取 `./.ai/api-status.yml`；前端和 AI 不得自行把接口标记为 `ready`。
- `planned`：不猜字段，等待契约或后端确认。
- `contracted/mockable`：可基于 OpenAPI 契约走统一 Mock 层。
- `partial`：稳定部分接真实 API，未完成部分继续统一 Mock，并在交付说明中标注。
- `ready`：默认切真实 API，并清理仅服务该接口的失效 Mock；暂不清理需说明原因和期限。
- `deprecated`：禁止新增依赖，已有调用提出替代或迁移方案。
- Mock 只能落在统一 Mock 层或统一 mock 目录，禁止写入 Vue 组件、Pinia action 或 API SDK 内部。

## 6. 前端交付

- 说明变更页面/组件、API 调用点、状态流转点和失败路径处理。
- 涉及接口调用时标注 `api-status` 状态、Mock 是否保留、真实 API 是否切换。
- 明确“已验证/未验证”；未执行真实验证时不得写“已完成/已修复/已通过”。

## 7. 何时联动

- 需要新增、删除、重命名接口字段或调整响应结构。
- 需要新增接口、合并接口、调整错误码或鉴权语义。
- 前端需求无法在既有契约下实现，需要后端 VO 或接口契约变更。
