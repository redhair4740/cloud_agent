# 跨模型适配层

本文件定义 AI_Vision 项目跨模型协作时的能力分类与降级策略，确保同一套 `.ai/` 规范体系在不同模型（GPT 系列 / Claude 系列 / Gemini 系列）下均可有效执行。

## 1. 模型能力分类

### 1.1 上下文窗口大小

| 等级 | 窗口范围 | 代表模型 | 适配说明 |
|------|----------|----------|----------|
| 大 | ≥ 128K tokens | GPT-4o / GPT-4.1 / Claude 3.5 Sonnet / Claude 3 Opus / Gemini 1.5 Pro / Gemini 2.5 Pro | 可完整加载全部 rules + agent.md + agents + skills，无需裁剪 |
| 中 | 32K–128K tokens | GPT-4 Turbo / Claude 3 Haiku / Gemini 1.5 Flash | 需按优先级裁剪 skills 和部分 agents，保留 rules + agent.md 完整 |
| 小 | < 32K tokens | GPT-3.5 Turbo / 早期小窗口模型 | 仅保留 rules 核心 + agent.md 摘要，agents 和 skills 全部裁剪 |

### 1.2 指令格式支持度

| 等级 | 支持能力 | 代表模型 | 适配说明 |
|------|----------|----------|----------|
| 高级 | 原生支持 XML 标签结构化指令、嵌套规则块、条件分支 | Claude 系列 / GPT-4o / GPT-4.1 / Gemini 2.5 Pro | 使用完整 XML 标签格式下发指令，保留全部结构化语义 |
| 中级 | 支持 Markdown 标题/列表/代码块结构，对 XML 标签理解不稳定 | GPT-4 Turbo / Gemini 1.5 Flash / GPT-3.5 Turbo | 将 XML 标签降级为 Markdown 标题 + 列表结构，保留层级但简化语法 |
| 基础 | 仅支持纯文本列表，复杂结构容易丢失或混淆 | 早期小模型 / 部分微调模型 | 将所有结构化指令降级为编号纯文本列表，去除嵌套 |

### 1.3 工具调用支持度

| 等级 | 支持能力 | 代表模型 | 适配说明 |
|------|----------|----------|----------|
| 原生支持 | 支持 function calling / tool_use 协议，可自动选择并调用工具 | GPT-4o / GPT-4.1 / Claude 3.5 Sonnet / Gemini 2.5 Pro | 直接使用原生工具调用协议，无需降级 |
| 部分支持 | 支持 function calling 但稳定性不足，或仅支持有限工具类型 | GPT-4 Turbo / Claude 3 Haiku / Gemini 1.5 Flash | 使用 function calling 但增加重试与校验，关键操作增加确认步骤 |
| 不支持 | 不支持 function calling，或调用失败率极高 | GPT-3.5 Turbo（部分版本）/ 早期模型 | 降级为手动操作指引，由模型输出操作步骤，人工确认执行 |

## 3. 指令格式降级规则

### 3.1 降级路径

```text
XML 标签格式 → Markdown 结构格式 → 纯文本列表格式
```

降级为单向不可逆：一旦确定目标模型能力等级，直接使用对应格式，不做中间态混合。

### 3.2 XML 标签格式（高级）

适用于指令格式支持度为"高级"的模型。

```xml
<rule name="repo-baseline" priority="1">
  <section name="沟通与证据表达">
    <constraint>全程使用简体中文沟通</constraint>
    <constraint>关键判断必须证据化：文件路径 + 方法/配置键 + 命令/参数 + 真实行为</constraint>
  </section>
</rule>
```

特征：
- 使用语义化 XML 标签表达规则层级与优先级
- 支持嵌套结构与条件分支
- 保留完整的元数据属性（优先级、名称、作用域）

### 3.3 Markdown 结构格式（中级）

适用于指令格式支持度为"中级"的模型。

```markdown
## 规则：仓库通用基线 [优先级: 1]

### 沟通与证据表达

- 全程使用简体中文沟通
- 关键判断必须证据化：文件路径 + 方法/配置键 + 命令/参数 + 真实行为
```

特征：
- 用 Markdown 标题层级替代 XML 嵌套
- 用列表替代约束标签
- 优先级等元数据放在标题行方括号内
- 去除 XML 标签，但保留层级语义

### 3.4 纯文本列表格式（基础）

适用于指令格式支持度为"基础"的模型。

```text
规则：仓库通用基线（优先级 1）

1. 沟通与证据表达
   1.1 全程使用简体中文沟通
   1.2 关键判断必须证据化：文件路径 + 方法/配置键 + 命令/参数 + 真实行为
```

特征：
- 使用编号列表替代所有结构化语法
- 层级通过编号缩进表达（1. / 1.1 / 1.1.1）
- 元数据放在括号内
- 去除所有 Markdown 和 XML 语法

### 3.5 降级转换对照

| 源格式元素 | XML | Markdown | 纯文本 |
|------------|-----|----------|--------|
| 规则块 | `<rule name="x">` | `## 规则：x` | `规则：x` |
| 优先级 | `priority="1"` | `[优先级: 1]` | `（优先级 1）` |
| 约束项 | `<constraint>` | `- 列表项` | `1.1 编号项` |
| 条件分支 | `<if>/<else>` | `> 条件说明` | `如果...则...否则...` |
| 代码示例 | `<code lang="x">` | ` ```x ` | 缩进 4 空格 |

## 4. 上下文长度适配策略

### 4.1 裁剪优先级

当模型上下文窗口不足以容纳全部 `.ai/` 规范内容时，按以下优先级从高到低保留：

```text
rules（最高，不可裁剪核心约束）
  > agent.md（项目总入口，保留核心章节）
    > agents（按任务类型选择性加载）
      > skills（最低，按需加载或裁剪）
```

### 4.2 各级别裁剪方案

#### 大窗口（≥ 128K）：完整加载

- 全部 rules 文件完整加载
- agent.md 完整加载
- 按任务类型加载对应 agents
- 按需加载相关 skills
- 无裁剪

#### 中窗口（32K–128K）：选择性裁剪

1. rules 文件完整加载，不做裁剪
2. agent.md 保留以下章节，裁剪示例与说明性段落：
   - 项目适用范围
   - 目录职责
   - 读取顺序与优先级
   - 规则组合继承与调度线
   - 任务类型与 Agent 选择
3. agents 仅加载当前任务类型对应的 1 个 agent，其余不加载
4. skills 仅在任务明确需要时加载 1 个相关 skill，其余不加载
5. 若裁剪后仍超限，对 agent.md 进一步压缩为摘要（仅保留章节标题与核心约束条目）

#### 小窗口（< 32K）：最小化加载

1. rules 仅保留当前任务类型对应的规则集合，其余不加载：
   - backend 任务：`00-repo-baseline.md + 10-backend-development-rules.md + 11-backend-object-layering-rules.md`
   - frontend 任务：`00-repo-baseline.md + 20-frontend-development-rules.md`
   - fullstack 任务：`00-repo-baseline.md + 10 + 11 + 20 + 30`
   - review 任务：按被评审对象加载对应规则集合
2. agent.md 压缩为最小摘要（仅保留目录映射 + 读取顺序 + 任务选择规则）
3. agents 不加载，将对应 agent 的核心职责条目内联到 agent.md 摘要中
4. skills 不加载
5. 若裁剪后仍超限，对 rules 文件逐条精简，仅保留"禁止"类硬约束条目

### 4.3 裁剪执行原则

- 裁剪只做删除或摘要，不做语义改写
- "禁止"类约束（不得/禁止/不允许）优先级高于"建议"类约束（推荐/建议/优先）
- 裁剪后必须保留的关键信息：模块边界、协作规则、验证口径、风险边界
- 裁剪决策应记录在对话上下文中，便于后续追溯

## 5. 工具调用降级策略

### 5.1 降级路径

```text
function calling（原生协议） → 带确认的 function calling → 手动操作指引
```

### 5.2 原生 function calling（原生支持）

适用于工具调用支持度为"原生支持"的模型。

行为：
- 模型通过原生 tool_use / function_call 协议自动选择并调用工具
- 调用结果自动回传，模型继续推理
- 无需人工干预

示例：
```json
{
  "tool_call": {
    "name": "read_file",
    "arguments": {"path": ".ai/agent.md"}
  }
}
```

### 5.3 带确认的 function calling（部分支持）

适用于工具调用支持度为"部分支持"的模型。

行为：
- 使用原生 function calling 协议发起调用
- 每次调用前增加确认步骤：模型先输出拟调用工具及参数，等待确认后再执行
- 对写操作（文件修改、命令执行）强制确认
- 对读操作（文件读取、搜索）可跳过确认，但增加重试机制
- 调用失败时自动重试 1 次，仍失败则降级为手动操作指引

确认格式：
```text
[工具调用确认] 即将调用：read_file({"path": ".ai/agent.md"})
请确认是否执行（Y/N）：
```

### 5.4 手动操作指引（不支持）

适用于工具调用支持度为"不支持"的模型。

行为：
- 模型不直接调用任何工具
- 改为输出结构化操作指引，由人工执行
- 每条操作指引包含：操作类型、目标、参数、预期结果
- 人工执行后将结果反馈给模型，模型继续推理

指引格式：
```text
[操作指引 #1]
类型：文件读取
目标：.ai/agent.md
预期：获取项目协作总入口内容
请执行后将内容粘贴到对话中。
```

```text
[操作指引 #2]
类型：文件修改
目标：src/main/java/com/example/Service.java
操作：将第 42 行的 `return null;` 替换为 `return Optional.empty();`
预期：消除空指针风险
请执行后确认结果。
```

### 5.5 操作类型映射

| 原生工具调用 | 手动操作指引 |
|-------------|-------------|
| `read_file(path)` | [操作指引] 类型：文件读取，目标：{path} |
| `write_file(path, content)` | [操作指引] 类型：文件修改，目标：{path}，操作：写入以下内容... |
| `search_code(query)` | [操作指引] 类型：代码搜索，关键词：{query} |
| `run_command(cmd)` | [操作指引] 类型：命令执行，命令：{cmd} |
| `grep(pattern, path)` | [操作指引] 类型：内容搜索，路径：{path}，模式：{pattern} |

### 5.6 降级触发条件

| 当前级别 | 触发降级条件 | 降级目标 |
|----------|-------------|----------|
| 原生 function calling | 连续 2 次调用返回解析错误 | 带确认的 function calling |
| 带确认的 function calling | 重试后仍失败，或模型无法正确构造调用参数 | 手动操作指引 |
| 手动操作指引 | 不再降级，维持当前级别 | — |

## 6. 适配执行流程

### 6.1 模型识别

1. 获取当前模型标识（model name / model id）
2. 对照第 2 节速查表确定三个能力维度等级
3. 若模型不在速查表中，按以下规则推断：
   - 上下文窗口：查询模型官方文档，按实际值归入大/中/小
   - 指令格式：先尝试 XML 格式，若响应中丢失结构则降级
   - 工具调用：先尝试原生 function calling，若失败则降级

### 6.2 适配执行

1. 根据上下文窗口等级执行裁剪策略（第 4 节）
2. 根据指令格式等级转换指令格式（第 3 节）
3. 根据工具调用等级选择调用方式（第 5 节）
4. 三项适配独立执行，互不阻塞

### 6.3 运行时降级

若适配后运行过程中出现以下情况，触发运行时降级：

- 指令格式降级：模型响应中反复丢失结构化约束或混淆层级
- 上下文降级：模型出现"遗忘"早期约束的行为，需进一步裁剪输入
- 工具调用降级：按第 5.6 节触发条件执行

运行时降级不回溯已执行的步骤，仅对后续交互生效。

## 7. 适配结果记录

每次跨模型适配执行后，应在对话上下文中记录：

- 当前模型标识与三个能力维度等级
- 实际执行的裁剪方案（保留了哪些文件/章节）
- 实际使用的指令格式等级
- 实际使用的工具调用等级
- 是否发生运行时降级及降级原因
