# 前端性能治理设计方案

## 1. 背景

- 需求来源：从旧前端瘦身方案中提炼当前仍可执行的治理口径。
- 当前现状：前端位于 `moon_cloud_frontend`，`package.json` 使用 Vue 3.5、Vite 5、Element Plus、Pinia、TypeScript。
- 触发原因：旧方案包含直接删除模块和卸载依赖的建议，当前需要按低风险顺序重整，避免破坏动态菜单、用户/角色、文件上传等基础能力。

## 2. 目标

| 编号 | 目标 | 验收指标 |
|------|------|----------|
| G-1 | 降低前端维护成本 | 清理非核心页面、菜单、API SDK 和专属组件 |
| G-2 | 降低构建和加载负担 | 源码引用清零后再卸载大依赖 |
| G-3 | 保住核心能力 | 登录、权限、用户、角色、菜单、字典、文件、算法、边缘模块不回归 |

## 3. 方案

### 3.1 保留清单

| 模块 | 原因 |
|------|------|
| `src/views/algorithm/**` | AI 视觉算法核心域 |
| `src/views/edge/**` | Edge 管理、监控、任务核心域 |
| `src/views/infra/file/**`、`src/views/infra/fileConfig/**` | 文件上传与文件存储基础能力 |
| `src/views/system/user/**` | 用户管理与登录后权限上下文 |
| `src/views/system/role/**` | 角色与菜单授权 |
| `src/views/system/menu/**` | 动态路由真源 |
| `src/views/system/dict/**` | 字典枚举真源 |
| `src/views/system/dept/**`、`src/views/system/post/**` | 用户管理、角色数据权限仍可能依赖，不能直接删除 |

### 3.2 删除优先级

第一优先级，低风险入口裁剪：

- `src/views/report/goview`
- `src/views/report/jmreport`
- `src/views/infra/apiAccessLog`
- `src/views/infra/apiErrorLog`
- `src/views/infra/server`
- `src/views/infra/skywalking`
- `src/views/infra/redis`
- `src/views/infra/druid`
- `src/views/infra/swagger`
- `src/views/infra/dataSourceConfig`

第二优先级，需要同步静态路由和菜单：

- `src/views/infra/codegen`
- `src/views/infra/job`
- `src/views/system/notify/my`
- `moon_cloud_frontend/src/router/modules/remaining.ts`
- 后端菜单数据与角色菜单授权。

第三优先级，需要先重构核心引用：

- `system/dept`
- `system/post`
- `system/notify`

这些模块只有在用户管理、角色数据权限、顶部消息入口和后端接口替代方案完成后才能删除。

### 3.3 依赖卸载顺序

依赖卸载必须放在源码引用清零之后。

| 依赖 | 前置清理 |
|------|----------|
| `bpmn-js`、`bpmn-js-token-simulation`、`diagram-js`、`dhtmlx-gantt` | 工作流/图形页面和专属组件无引用 |
| `@form-create/designer`、`@form-create/element-ui` | 表单设计器、`plugins/formCreate`、`utils/formCreate` 无引用 |
| `@wangeditor-next/*` | 富文本组件无引用 |
| `markdown-it`、`markmap-*` | Markdown/思维导图组件无引用 |
| `echarts-wordcloud` | 词云页面和组件无引用 |

`echarts` 本体不纳入第一轮卸载，因为监控、首页或通用图表仍可能使用。

## 4. 影响范围

- 前端：`moon_cloud_frontend/src/views/**`、`src/api/**`、`src/router/modules/remaining.ts`、`package.json`。
- 后端：菜单接口、权限数据、可能关联的 Controller/Service/SQL。
- 数据库：菜单表和角色菜单授权需要同步清理，属于高风险数据操作，必须另行确认。
- 配置/部署：构建缓存和依赖锁文件会变化，卸载依赖前需确认执行窗口。

## 5. 验收标准

- [ ] 功能验收：登录、动态路由、用户、角色、菜单、字典、文件、算法、边缘管理、边缘监控、边缘任务页可访问。
- [ ] 契约验收：后端菜单不再下发已删除页面的 `component` 路径。
- [ ] 验证要求：执行 `pnpm ts:check`、`pnpm build:local`，并浏览器验证核心页面。

## 6. 风险与回滚

- 风险：只删页面不清菜单，会导致动态路由 `Cannot find module` 或白屏。
- 风险：先卸载依赖会导致仍有源码引用时报编译错误。
- 风险：直接删除部门、岗位、通知会影响用户管理、角色数据权限和顶部消息入口。
- 回滚方案：保留独立提交；若构建失败，先回滚删除的页面/API/依赖变更，再重新按引用清零顺序处理。
- 未验证项：本次未执行前端构建、未跑动态路由、未扫描所有依赖引用。
